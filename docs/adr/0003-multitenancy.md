# ADR-0003. Мультитенантность: tenant_id + автофильтрация + RLS

- **Статус:** принято (Фаза 0/1, 2026-07-06; источник — мастер-промпт v2 №3; уточнено решениями ОВ-01, ОВ-03, ОВ-04)
- **Контекст.** Кросс-тенантная утечка (IDOR/BOLA) — самый вероятный и самый дорогой инцидент мультитенантного SaaS (threat model V1). Одна линия обороны недостаточна: забытый фильтр в одном запросе = утечка.
- **Решение.** Две независимые линии:
  1. **Repository (первая линия, код).** Все бизнес-таблицы наследуют `TenantScopedBase` (uuid7 PK + `tenant_id NOT NULL`); `shared.repository.Repository` строит каждый запрос через единственную точку с `WHERE tenant_id = ctx.tenant_id` и штампует tenant_id на INSERT. Обойти фильтр из наследника нельзя — его там просто нет.
  2. **RLS в Postgres (вторая линия, БД).** Транзакционно-локальный контекст `SET LOCAL app.tenant_id` + политики на каждой тенантной таблице; непривилегированные runtime-роли. Механика — схема §3; реализуется вместе с первыми тенантными таблицами в **Фазе 2** (в Фазе 1 тенантных таблиц нет).
- **Утверждённые уточнения (Фаза 0).** ОВ-01: `users` и auth-таблицы — глобальные (одна личность — много организаций); ОВ-03: tenant попадает в контекст клеймом JWT (токен на пару user×tenant); ОВ-04: платформенная роль — флаг вне тенантов.
- **Последствия.** Каждая новая таблица обязана иметь tenant_id + тест изоляции (DoD); системные пути без тенанта — только чтение через `SystemRepository` (импорт разрешён только core/, контракт import-linter) с последующим повышением контекста.
