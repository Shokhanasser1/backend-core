# ADR-0006. Событийная шина: in-process pub/sub + reliable-доставка через arq

- **Статус:** принято (Фаза 0/1, 2026-07-06; источник — мастер-промпт v2 №6; контракт — интерфейсы §2.6; уточнено ОВ-09, ОВ-38)
- **Контекст.** Модули должны реагировать друг на друга, не зная друг о друге: commerce публикует `commerce.order.created` — будущий crm подписывается. Полноценный брокер сообщений для этого масштаба избыточен.
- **Решение.**
  - Имена событий — `<module>.<entity>.<action>` с префиксом модуля (ОВ-09); формат проверяется шиной при publish/subscribe.
  - Публикация — **только после commit** (`Service.emit` копит события, UnitOfWork публикует post-commit hook'ом). Rollback = события отброшены.
  - `reliable=False` — in-process, at-most-once, ошибка обработчика логируется и не роняет издателя. `reliable=True` — джоба arq, at-least-once с дедупликацией: `INSERT (handler, event_id) ON CONFLICT DO NOTHING` в таблицу `processed_events` в одной транзакции с UoW обработчика; 5 попыток с экспоненциальным backoff, затем dead letter (лог + Sentry).
  - Wildcard/префиксные подписки — привилегия core-модулей (v1: audit); фичи подписываются только на явные имена из своих `listens_events`.
- **Ограничение v1 (ОВ-38, принято).** Post-commit-публикация: краш процесса между commit и постановкой arq-джобы теряет событие. Поэтому внутримодульные критичные реакции через шину не ходят (активация подписки — в транзакции финализации платежа). Transactional outbox — бэклог; конверт события совместим, переход не потребует миграций.
- **Последствия.** Регистрация подписок обязана лежать в модулях, импортируемых и web-, и worker-процессом; in-process подписчик видит только события своего процесса — межпроцессная доставка только через `reliable=True`.
