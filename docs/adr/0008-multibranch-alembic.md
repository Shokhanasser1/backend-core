# ADR-0008. Мультиветочный Alembic: миграции в папках компонентов

- **Статус:** принято (решение владельца 2026-07-06, ОВ-10; источник — стратегия обновления §3.5, §4.3)
- **Контекст.** Ядро выпускает новую миграцию, у клиентского проекта — свои. При линейной истории Alembic это гарантированные «две головы» и ручной `alembic merge` в каждом проекте на каждый релиз шаблона.
- **Решение.**
  - У каждого компонента — своя ветка ревизий с branch label: `shared` (служебные таблицы шины), далее `core_auth`, `core_tenants`, … (Фаза 2+), `commerce_products`, … (Фаза 6+); клиентский проект ведёт свои миграции в ветке `project`.
  - Ревизии лежат в папке компонента (`shared/migrations/`, `core/auth/migrations/`, …); корневой `migrations/` — только `env.py`, конфиг и обёртка.
  - `version_locations` собирается **динамически** сканированием дерева (`migrations/discovery.py`). Alembic читает version_locations до выполнения env.py, поэтому запуск — через обёртку `python -m migrations.cli` (= `make migrate`); статический список в alembic.ini давал бы конфликты merge при каждом добавлении фичи.
  - Накат — всегда `alembic upgrade heads` (множественное число); тест «после наката не осталось неприменённых голов» — в CI. Межветочные зависимости — через `depends_on`.
  - Каждая миграция обратима (рабочий downgrade) — проверяется тестом.
- **Отложено до Фазы 2.** Роли БД (`app_migrator`/`app_user`/`app_maintenance`/`app_retention`) — объекты кластера, создаются bootstrap-скриптом окружения; GRANT'ы и RLS-хелперы (`enable_tenant_rls`) приходят с базовой миграцией ядра в Фазе 2 (схема §3.6). До этого dev-окружение работает одной ролью compose.
- **Последствия.** Новая миграция ядра приезжает merge'ем, продолжает свою ветку и не пересекается с клиентской цепочкой — конфликтов графа ревизий нет by design. Изменение схемы таблиц шаблона в клиентском проекте запрещено даже как помеченная кастомизация; правило контролирует джоба template-drift (проверка «в»).
