# ADR-0009. Стратегия обновления шаблона: git-шаблон + документированный merge

- **Статус:** принято (Фаза 0, утверждено владельцем 2026-07-06; полный анализ — `docs/phase0/04-update-strategy.md`)
- **Контекст.** Фикс бага в ядре (включая security) должен доезжать до всех активных клиентских проектов за предсказуемое время (целевой SLA security-фиксов — 24–48 ч) силами соло-разработчика; v1 рассчитан на 1–10 проектов.
- **Решение (v1 — вариант А).** Клиентский проект создаётся только `git clone` шаблона (remote `template` сохраняется); обновление — `git merge --no-ff vX.Y.Z` по процедуре `docs/UPDATE.md`. Ключевые конвенции (§4.2 стратегии): merge только тегов настоящим merge-коммитом (squash/rebase запрещены); шаблонные пути (`core/`, `shared/`, приехавшие фичи в `modules/`) в клиентском проекте read-only, неизбежные правки помечаются `# CUSTOM(<project>)` и декларируются в `CUSTOMIZATIONS.md`; SemVer + CHANGELOG (keep-a-changelog) с Upgrade notes; `[SECURITY]`-фиксы — минимальными изолированными коммитами.
- **Механика в этом репозитории (Фаза 1).** Константа `shared.TEMPLATE_VERSION` (обновляется релизом, логируется на старте, наружу не публикуется); CHANGELOG.md; скелет `docs/UPDATE.md`; заготовка CI-джобы `template-drift` (три проверки: общий предок, незадекларированный дрейф, владение Alembic-ветками); мультиветочный Alembic (ADR-0008); `make setup` включает `git config rerere.enabled true`.
- **Эволюция.** Переход на copier — по явным триггерам (≥4–5 активных проектов; раскатка релиза регулярно >2 ч; спрос на scaffolding; два подряд релиза с конфликтами структуры); путь миграции описан в стратегии §4.4. Вариант «ядро как пакет» — целевое состояние на «десятках» проектов.
