# ADR-0010. Алгоритм подписи JWT: HS256 в v1, план миграции на асимметрию

- **Статус:** принято (решение владельца 2026-07-06, ОВ-17)
- **Контекст.** Ядро v1 — монолит с единственным владельцем секрета подписи. Будущий sibling-шаблон tg-bot (бэклог) будет проверять токены этого же ядра — тогда понадобится верификация без общего секрета (асимметрия).
- **Решение (v1).** Access-токен — JWT **HS256** (симметричный). Секрет — только из окружения (`JWT_SECRET`). **Верификатор принимает строго один алгоритм из конфига** (`settings.jwt_algorithm`), а не из заголовка `alg` самого токена. Это закрывает класс alg-confusion: подстановку `alg:none` и — при будущей миграции на RS256 — верификацию RS256-токена как HS256 с публичным ключом в роли HMAC-секрета (threat model V3). Токен несёт клейм `tenant` (пара user×tenant, ОВ-03) и `jti`.
- **План миграции на асимметрию.** Когда появятся внешние верификаторы (tg-bot): перейти на RS256/EdDSA. Обязательные условия: (1) верификатор по-прежнему принимает **строго ожидаемый** алгоритм из конфига (иначе публичный ключ можно скормить как HMAC-секрет); (2) окно ротации с двумя ключами (kid в заголовке для выбора ключа проверки); (3) приватный ключ — только у эмитента (ядра), публичный — у верификаторов. Access-токен уже несёт всё нужное; переход не ломает подписчиков.
- **Последствия.** В v1 симметричный секрет разделяют все, кто проверяет токены (в монолите — только ядро); приемлемо. Refresh-токены — непрозрачные случайные строки, хранятся хэшем, к алгоритму JWT отношения не имеют.
