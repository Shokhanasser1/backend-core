# Сверка платежей (reconciliation)

> Статус: **ручная процедура (v1)** — решение ОВ-23. Автоматическая сверка
> (фоновая arq-задача, сверяющая наши статусы с API/выписками провайдеров) —
> в бэклоге.

## Зачем это нужно

Итог платежа мы узнаём из вебхука провайдера (Payme/Click). Вебхук может
**не дойти** (сетевой сбой, даунтайм нашего сервиса дольше окна ретраев
провайдера) или разойтись с реальностью из-за инцидента на стороне провайдера.
Тогда наш `payments.status` расходится с фактом у провайдера: деньги списаны, а
подписка/заказ не активированы (или наоборот). Reconciliation — регулярная
ручная сверка, которая ловит такие расхождения.

Внутренняя защита уже стоит (threat model V4): подпись, привязка к тенанту
только из наших записей, идемпотентность по `(provider, dedup_key)`, сверка
суммы/валюты, машина состояний. Reconciliation закрывает то, что защита в
принципе поймать не может — **молчаливую недоставку** и **сбой провайдера**.

## Периодичность

- Ежедневно — беглый просмотр «зависших» платежей (below).
- Ежемесячно — полная сверка с выпиской провайдера (Payme Business / кабинет
  Click) за период.

## Что сверять

### 1. Зависшие платежи (наши данные)

Платежи, застрявшие в незавершённом статусе дольше TTL протухания
(`payment_checkout_ttl_seconds`, по умолчанию 1 час) — джоба протухания должна
была перевести их в `expired`; если такие остались, джоба не отработала:

```sql
SELECT id, tenant_id, provider, provider_transaction_id, amount, currency,
       status, created_at
FROM payments
WHERE status IN ('created', 'pending')
  AND created_at < now() - interval '1 hour'
ORDER BY created_at;
```

Для каждого — проверить в кабинете провайдера фактический статус транзакции
(по `provider_transaction_id` или нашему `id` = merchant order id).

### 2. Расхождение «оплачено у провайдера ≠ succeeded у нас»

По выписке провайдера за период выбрать успешные транзакции и сверить, что у нас
соответствующий `payments.status = 'succeeded'`, сумма и валюта совпадают,
подписка/заказ активированы. Ключ сопоставления:

- по нашему `payment_id` (merchant order id, он же `ac.payment_id` у Payme /
  `merchant_trans_id` у Click);
- либо по `provider_transaction_id`.

### 3. Журнал вебхуков

Отклонённые и неудачно обработанные вебхуки — источник для разбора инцидента:

```sql
SELECT provider, dedup_key, status, error, created_at
FROM payment_webhooks
WHERE status IN ('rejected', 'failed')
ORDER BY created_at DESC;
```

`rejected` с `dedup_key` вида `rejected:<hash>` — трафик с неверной подписью
(норма при сканировании/атаках, но всплеск = повод проверить ротацию ключа).

## Что делать при расхождении

Финализировать платёж **только через штатный путь** — повторную/ручную доставку
вебхука провайдеру (он переиграет колбэк, обработчик отработает идемпотентно и
переведёт статус в той же транзакции, что и активацию подписки). **Не** править
`payments.status` руками в БД: прямой UPDATE обойдёт активацию подписки и аудит и
создаст рассинхрон «оплата есть — подписка не активна».

Если провайдер списал деньги по транзакции, которой у нас нет платежа (вебхук по
неизвестному платежу — в логах `payment webhook for unknown payment`), — это
аномалия: разобрать вручную, при необходимости вернуть деньги (возвраты в v1 —
ручная операция вне системы, ОВ-22).

## Ограничения v1

- Возвраты (refund) не поддерживаются (ОВ-22): `succeeded` терминален. Возврат
  покупателю — ручная операция вне системы.
- Автоматической сверки нет: расхождение из-за недоставки вебхука ловится только
  этой процедурой.
