> Статус: утверждена владельцем 2026-07-06 (реестр решений —
> `00-open-questions.md`; решённый пункт §5 №6 аннотирован). Источник

> После утверждения оформляется как ADR в Фазе 1 — вместе с ADR
> «Мультиветочный Alembic» (см. §3.5 и открытый вопрос №6).

## 1. Постановка задачи

**Кто эксплуатирует.** Соло-разработчик / микро-команда (1–3 человека). Нет
выделенного платформенного инженера, нет бюджета на тяжёлую инфраструктуру.
Любая процедура, требующая больше ~15 минут ручной работы на проект,
на горизонте десятков проектов перестаёт выполняться — это главный риск,
и он прямо учтён в требовании №3 ниже.

**Как рождается клиентский проект.** Проект собирается из шаблона
backend-core (клонирование репозитория, см. конвенцию 4.2.1) и, при
необходимости, переносом отдельных фич инструментом `tools/add-feature`.
Дальше проект кастомизируется под клиента: свои фичи, конфигурация, шаблоны
уведомлений, иногда — правки поведения ядра.

**Что требуется от стратегии.**

1. Фикс бага в ядре (включая security-фикс) доставляется во все активные
   клиентские проекты за предсказуемое время. **Целевой** SLA для критичных
   security-фиксов — прод всех активных проектов в течение 24–48 часов.
   SLA помечен целевым сознательно: его экономическое основание (модель
   поддержки) и само определение «активного проекта» — открытые вопросы
   №3 и №4; до их решения это ориентир для проектирования механики,
   а не договорное обязательство.
2. Обновление не ломает кастомизации клиентских проектов и не требует
   их переписывания.
3. Масштаб: **v1 рассчитан на 1–10 активных проектов.** Рост к «десяткам» —
   явная точка пересмотра стратегии (см. 4.4, «Судьба варианта Б»): на таком
   масштабе любая ручная процедура пробивает бюджет из абзаца «Кто
   эксплуатирует» даже в бесконфликтном случае, и честный ответ там —
   полностью автоматическая доставка, а не ускорение ручной. До точки
   пересмотра допустима эволюция инструмента, недопустима смена модели
   данных «что где лежит».
4. Дрейф версий наблюдаем на двух уровнях: «фикс **влит** в репозиторий
   проекта» — машиночитаемо из самого репозитория; «фикс **в проде**» —
   по реестру проектов (отметка после deploy; merge ≠ deploy, см. 3.7).

**Термины.**

- *Релиз шаблона* — тегированное состояние backend-core (`vX.Y.Z`).
- *Клиентский проект* — репозиторий, собранный из шаблона.
- *Шаблонные пути* — `core/`, `shared/` и фичи, приехавшие из шаблона
  в `modules/`.
- *Дрейф* — накопившаяся незадекларированная разница между клиентским
  проектом и последним влитым релизом шаблона.

## 2. Варианты

### 2.1 Вариант А — git-шаблон + документированный merge

Клиентский проект создаётся как `git clone` шаблона (с полной историей),
remote шаблона сохраняется под именем `template`, рабочий origin —
репозиторий проекта. Доставка фикса: в шаблоне выпускается тег `vX.Y.Z`;
в каждом проекте выполняется `git fetch template && git merge --no-ff vX.Y.Z`
по документированной процедуре, затем миграции и тесты.

- Поток фикса: релиз в шаблоне → merge тега в каждом проекте → `make test`
  → deploy.
- Кастомизации живут в истории клиентского проекта; git делает трёхстороннее
  слияние от общего предка (последнего влитого тега).

Плюсы: нулевая инфраструктура; весь код виден и правится в проекте;
конфликты решаются штатным механизмом git по месту. Минусы: обновление —
ручная операция на каждый проект; дисциплина (clone, а не копия файлов;
merge, а не squash) требует защиты — в v1 её даёт не только конвенция,
но и CI-проверка предка (4.2.6); дрейф не виден без собственного учёта.
`git rerere` облегчает повторные разрешения конфликтов, но только локально
на машине, где конфликт уже решали (см. 3.2), — как свойство репозитория
его учитывать нельзя.

### 2.2 Вариант Б — ядро как приватный пакет

`backend-core` собирается в пакет и публикуется (минимально —
git-зависимость uv по тегу: `backend-core = { git = "...", tag = "v1.4.2" }`;
максимально — приватный индекс). Клиентский проект зависит от версии пакета;
`core/` и `shared/` физически лежат в site-packages и не редактируются.
Кастомизации — только снаружи пакета: свои фичи, конфигурация, точки
расширения, которые пакет обязан предоставить.

- Поток фикса: релиз пакета → bump версии в `pyproject.toml` + `uv lock`
  в каждом проекте → CI → deploy. Почти полностью скриптуется.

Плюсы: самая дешёвая и автоматизируемая доставка; дрейф читается из
lockfile; редактирование ядра в проекте невозможно физически. Минусы:
кастомизация внутри ядра невозможна без форка пакета (форк убивает дешёвые
обновления) или monkey-patch (запрещаем); требует спроектировать полный
набор точек расширения ядра заранее — прямое противоречие правилу
«не изобретай абстракции на будущее»; раскалывает конструктор на две модели
(«ядро — пакет, фичи — копируемые папки»); нужна инфраструктура сборки
и публикации пакета. (Релизная дисциплина — теги, SemVer, CHANGELOG —
минусом Б не считается: она нужна всем вариантам и входит в общий базис,
см. §3 и конвенцию 4.2.4.)

### 2.3 Вариант В — copier

Шаблон оформляется как copier-template (`copier.yml`, ответы генерации:
имя проекта, `ENABLED_MODULES` и т.п.). Проект создаётся командой
`copier copy`, версия шаблона фиксируется в `.copier-answers.yml`
(поле `_commit`). Обновление — `copier update`: copier вычисляет diff между
старой и новой версией шаблона и накладывает его трёхсторонним слиянием
на проект; конфликты — inline-маркерами (copier ≥ 9).

Плюсы: инструмент создан ровно под эту задачу; версия шаблона в проекте
машиночитаема из коробки; генерация решает и scaffolding (переименования,
выбор модулей при создании). Минусы: шаблон обрастает Jinja-разметкой хотя бы
в части файлов; нужен CI-тест генерации; новая семантика для команды;
поведение при конфликтах в итоге сопоставимо с git merge — чудес нет;
`copier-scaffolding` явно вынесен в бэклог, строить его в v1 нельзя —
здесь он рассматривается только как направление эволюции стратегии.

### 2.4 Осмысленные гибриды

- **А → В (эволюция во времени).** Начать с git-merge, перейти на copier,
  когда проектов станет много. Совместимость высокая: обе модели держат
  полный исходник в проекте и опираются на конвенцию «шаблонные пути
  не редактируются»; конвенции v1 (теги, CHANGELOG, Alembic-ветки, реестр)
  переносятся без потерь. Важная оговорка: адопция существующих проектов
  **не бесплатна** — `copier update` требует, чтобы на базовой версии
  (`_commit`) шаблон уже был copier-шаблоном (содержал `copier.yml`),
  поэтому теги, выпущенные до копиеризации, базой служить не могут.
  Реальный путь адопции: выпустить первый copier-совместимый релиз →
  довезти его во все проекты обычным git-merge → только он становится
  `_commit`-базой. Подробно — в 4.4.
- **Б + копируемые фичи (раскол по слоям).** Ядро — пакет, фичи — папки.
  Архитектурно чисто, но требует зрелых точек расширения ядра, которых
  в v1 нет и быть не должно. Рассматривается только как поздняя эволюция.
- Гибрид А+Б («часть ядра пакетом») отвергается сразу: две процедуры
  обновления вместо одной — худшее из обоих миров для соло-разработчика.

## 3. Критерии сравнения

Общий базис всех трёх вариантов (в сравнении не участвует, входит
в конвенции 4.2): релизная дисциплина шаблона — теги `vX.Y.Z`, SemVer,
`CHANGELOG.md` с Upgrade notes. Различия ниже — сверх этого базиса.

| Критерий | А. git-шаблон + merge | Б. ядро-пакет | В. copier |
|---|---|---|---|
| Доставка одного фикса в 10 проектов | без конфликтов ~2–3 ч; с типичной долей конфликтов — до полудня; полуавтомат | ~0.5–1 ч, автоматизируется полностью | ~1.5–2 ч, полуавтомат; конфликты ручные |
| Конфликт с кастомизацией внутри ядра | штатный git-конфликт, решается по месту; повторы гасятся правилом выноса кастомизации (3.2); rerere — локальное удобство, не гарантия | конфликт невозможен: кастомизация запрещена — либо upstream в шаблон, либо форк (теряются обновления) | inline-маркеры, сопоставимо с git; повторные конфликты не кешируются |
| Инфраструктура сверх базиса | процедура в docs + CI-джоба `template-drift` | сборка и публикация пакета, релизный конвейер | `copier.yml`, Jinja в части файлов, CI-тест генерации |
| Анатомия фич и `tools/add-feature` | без изменений: фича = папка | раскол: ядро в site-packages, фичи копиями; двойная модель обновления | без изменений; файлы, добавленные проектом, `copier update` не трогает |
| Миграции Alembic | ветки по компонентам (3.5), новые миграции приезжают merge'ем | те же ветки; `version_locations` из установленного пакета | как в А |
| Кривая обучения | низкая, но с фатальными ошибками новичка (squash/rebase, правка ядра без маркера) — гасятся CI-проверками (3.6) | средняя: релизный цикл, версии, публикация | средняя+: новый инструмент, семантика update, шаблонизация |
| Риск дрейфа и его мониторинг | высокий без учёта; компенсации: `TEMPLATE_VERSION` + реестр (3.7) | минимальный: версия в lockfile | низкий: `_commit` в `.copier-answers.yml` |
| Тестируемость обновления | CI проекта после merge; репетиция на эталонном проекте | CI проекта после bump; возможна матрица версий | CI проекта после update; плюс CI-тест генерации |

### 3.1 Трудоёмкость доставки фикса

- **А:** на проект — fetch, merge тега, push тега в origin (нужен джобе
  `template-drift`, см. 4.2.6), `make migrate` (= `alembic upgrade heads`,
  см. 3.5), `make test`, push, deploy. Чистый merge: 10–15 минут; конфликт
  добавляет 15–60 минут. Итого на 10 проектов: без конфликтов ~2–3 часа,
  с типичной долей конфликтов — до полудня (~4 ч). В целевой SLA 24–48 ч
  укладывается. Частично скриптуется (цикл по реестру), но разрешение
  конфликтов всегда ручное.
- **Б:** на проект — правка одной строки в `pyproject.toml`, `uv lock`,
  push, CI. 3–5 минут, полностью скриптуется; человек нужен только
  на красный CI.
- **В:** на проект — `copier update --defaults` + тесты. Между А и Б;
  конфликты, как и в А, ручные.

Честное следствие для масштаба: на «десятках» проектов А **и** В пробивают
бюджет «~15 минут ручной работы на проект» даже без конфликтов. Требование
№3 (§1) поэтому ограничивает v1 десятью проектами, а масштаб «десятки»
объявлен точкой пересмотра в пользу Б (4.4) — это не дефект рекомендации,
а её явная граница.

### 3.2 Конфликты с кастомизациями внутри ядра

Ключевой сценарий рынка: «клиенту нужно чуть иначе», и не всё сводится
к конфигу. Правда в том, что **ни один вариант не сливает конфликт
автоматически** — разница в том, где и как конфликт проявляется:

- **А** — конфликт в момент merge, в контексте файла, с историей «зачем
  клиент это менял» в git log. Худшее свойство: тот же файл конфликтует
  на каждом релизе. **Главный гаситель повторных конфликтов — правило:
  при первом же конфликте в шаблонном пути кастомизация выносится из ядра**
  (в свою фичу-папку, конфигурацию, обработчик события) либо уходит
  upstream в шаблон. `git rerere` — только локальное удобство: его кеш
  (`.git/rr-cache`) живёт в конкретном клоне, не версионируется, не
  приезжает с clone и не существует ни у подрядчика на свежей машине,
  ни в CI. Он включается bootstrap-скриптом (`make setup` пишет
  `git config rerere.enabled true`), но закладываться на него как на
  механизм стратегии нельзя.
- **Б** — конфликтов нет, потому что запрещена сама кастомизация. Каждая
  клиентская хотелка внутри ядра превращается в задачу «обобщи и выпусти
  релиз шаблона» — для соло-разработчика это блокер скорости поставки.
- **В** — как А, но повторные конфликты не гасятся даже локально; плюс
  возможны ложные конфликты от Jinja-разметки, если она попала в горячие
  файлы.

### 3.3 Инфраструктура

Релизная дисциплина (теги, SemVer, CHANGELOG) нужна всем вариантам
одинаково — она вынесена в базис и в сравнении не участвует. Сверх базиса:
А не требует ничего, кроме существующего git-хостинга и одной CI-джобы
в клиентском проекте. Б требует сборку пакета и место публикации;
git-зависимость uv по тегу снимает вопрос индекса, но не снимает сборку
и отладку пакетирования. В требует поддерживать шаблонность репозитория
и тестировать генерацию в CI.

### 3.4 Влияние на анатомию фич и tools/add-feature

Анатомия «фича = самодостаточная папка, переносится копированием» —
несущая стена конструктора. А и В её не трогают. Б её раскалывает:
фичи модулей остаются копиями в проекте, а ядро уезжает в пакет — появляются
две разные механики обновления в одном проекте, и README/инструкции
приходится писать под обе.

Важное следствие для всех вариантов: **автодискавери** (фичи, admin-экраны,
роутеры собираются сканированием папок, а не правкой центральных файлов
регистрации) — главный гаситель конфликтов. Клиентская фича = новая папка,
merge шаблона её не касается. Это уже заложено в архитектуру (feature.toml,
автодискавери на старте) и обязано сохраниться в реализации Фаз 1 и 6.

### 3.5 Миграции Alembic

Сценарий: ядро выпустило новую миграцию auth, а у клиентского проекта есть
свои миграции. При линейной истории Alembic это гарантированная беда:
две головы и ручной `alembic merge` в каждом проекте на каждый релиз.

Решение (закладывается в Фазе 1, одинаково работает для А, Б и В) —
**мультиветочный Alembic**:

- у каждого компонента своя ветка ревизий с `branch_labels`:
  `core_auth`, `core_tenants`, `core_billing`, `core_notifications`,
  `core_audit`, далее по фиче (`commerce_products`, ...);
- миграции каждого компонента лежат в его папке (`core/auth/migrations/`,
  `modules/commerce/products/migrations/` — для фич это уже зафиксировано
  анатомией в мастер-промпте), а `version_locations` собирается динамически
  из папок присутствующих компонентов. **Это отклонение от структуры
  мастер-промпта**, где показан корневой каталог `migrations/`: он
  вырождается в `env.py` + конфигурацию Alembic. Отклонение вынесено
  на утверждение отдельно — открытый вопрос №6 — и фиксируется в ADR
  «Мультиветочный Alembic» (Фаза 1);
- клиентский проект ведёт свои миграции в собственной ветке `project`
  и никогда не дописывает ревизии в чужие ветки; правило контролируется
  не глазами, а CI — проверкой владения ветками в составе джобы
  `template-drift` (4.2.6): каждая ревизия вне ветки `project` лежит
  в шаблонном пути и её `down_revision` принадлежит той же ветке; ревизии
  ветки `project` живут только в папке проекта;
- межветочные зависимости (например, FK клиентской таблицы на таблицу ядра)
  объявляются через `depends_on` на конкретную ревизию;
- накат — всегда `alembic upgrade heads` (множественное число). Цель
  `make migrate` обязана вызывать именно `upgrade heads`; в CI — тест
  «после наката не осталось неприменённых голов» (классическая ошибка —
  `upgrade head` в единственном числе, молча накатывающий одну ветку).

Тогда новая миграция ядра приезжает (merge'ем в А/В, bump'ом версии в Б),
продолжает свою ветку и не пересекается с клиентской цепочкой:
**конфликтов графа ревизий нет by design**. Семантические конфликты ветки
не лечат: клиентская правка схемы таблицы ядра столкнётся с приехавшей
миграцией ядра на ту же таблицу. Поэтому изменение схемы таблиц шаблона
в клиентском проекте запрещено даже как помеченная кастомизация
(конвенция 4.2.3) — клиенту разрешены только собственные таблицы в ветке
`project` с FK через `depends_on`. Требование обратимости миграций
(downgrade) не меняется.

### 3.6 Кривая обучения

А — низкая, но не нулевая: сама процедура — одна страница поверх обычного
git merge, однако существуют фатальные ошибки новичка: `merge --squash`
или rebase уничтожают общего предка (каждый следующий релиз конфликтует
заново), правка ядра без маркера — тихий дрейф. Ответ стратегии — не память
людей, а автоматика: проверка предка и сверка дрейфа в CI (4.2.6) превращают
эти ошибки в красный CI, а не в отложенную катастрофу. Б — релизная
дисциплина пакета и отладка «почему в site-packages не то». В — сам copier
прост, но семантику update и границы «что перегенерируется, что нет» нужно
один раз честно выучить; учить придётся и будущих подрядчиков.

### 3.7 Дрейф версий и мониторинг

Наблюдаемость — двухуровневая (требование №4): «фикс влит» и «фикс в проде».

- **А:** дрейф — главный риск варианта. Компенсации: константа версии
  шаблона в коде (`shared/__init__.py: TEMPLATE_VERSION`), обновляемая
  релизом, приезжающая merge'ем и логируемая при старте приложения (наружу
  не публикуется — только логи и закрытая админ-диагностика); реестр
  клиентских проектов (4.2.7); read-only скрипт по реестру, читающий
  `TEMPLATE_VERSION` из веток main, — он отвечает **только** на вопрос
  «фикс влит». Вопрос «фикс в проде» закрывает единственный источник
  правды — отметка в реестре после deploy (шаг процедуры 4.2.8), потому
  что merge ≠ deploy. Шаблон реестра и скрипт — работа Фазы 5 (4.3).
- **Б:** версия в `uv.lock` — «влит» виден grep'ом; «в проде» — так же
  через реестр. Лучший вариант по этому критерию.
- **В:** `_commit` в `.copier-answers.yml` — машиночитаемо, как в Б;
  «в проде» — так же через реестр.

### 3.8 Тестируемость обновления

Во всех вариантах последний рубеж один — **CI клиентского проекта**:
обновление считается вставшим, когда после merge/bump/update зелёные
`make lint`, `make test` (с testcontainers-Postgres) и `alembic upgrade
heads` на копии структуры БД. Различия:

- А/В: перед раскаткой по проектам фикс репетируется на «эталонном
  клиентском проекте» (см. 4.3, Фаза 6) — дешёвая страховка процедуры;
- Б: дополнительно возможна матрица «пакет × проекты» в CI, но для
  соло-масштаба это роскошь;
- security-фикс во всех вариантах закрывается чек-листом по реестру:
  проект → merge/bump → CI зелёный → deploy → отметка в реестре;
  для отставших проектов действует hotfix-исключение (4.2.8).

## 4. Рекомендация

### 4.1 Решение

**V1: вариант А — git-шаблон + документированный merge, с жёсткими
конвенциями, CI-защитой конвенций и минимальным инструментальным обвесом.
Запланированная эволюция — на copier (вариант В) по явным триггерам;
copier-scaffolding остаётся в бэклоге и в v1 не строится.** Вариант Б
(пакет) отклоняется для масштаба v1, но зафиксирован как целевое состояние
при выходе на «десятки» проектов или стабилизации ядра (4.4).

Обоснование:

1. **Экономика соло-разработчика на масштабе v1 (1–10 проектов).**
   Стоимость доставки фикса через merge мала, стоимость владения
   инфраструктурой — нулевая. Пакет окупается только при стабильном ядре
   и полном отказе от кастомизаций внутри него; в первый год жизни шаблона
   ядро будет меняться после каждого «боевого» проекта (см. PLAN.md,
   «После V1 — проверка боем»). На «десятках» проектов баланс меняется —
   это честно зафиксировано как граница рекомендации (§1 требование №3,
   §3.1, 4.4).
2. **Кастомизации внутри ядра неизбежны на этом рынке** — вариант Б делает
   их запрещённо дорогими, А и В — управляемыми. При этом конвенция
   «шаблонные пути read-only» (4.2.3) удерживает их количество около нуля.
3. **А и В лежат на одной эволюционной прямой**: обе модели — «полный
   исходник в проекте + слияние изменений шаблона». Всё, что закладывается
   под А (теги, SemVer, CHANGELOG, Alembic-ветки, автодискавери, реестр),
   дословно переиспользуется при переходе на В. Оговорка: адопция в copier
   требует довезти первый copier-совместимый релиз обычным merge'ем (2.4,
   4.4) — это дёшево, но не бесплатно; «ничего не сжигается», но один
   переходный релиз обязателен.
4. **Бэклог уже указывает направление**: copier-scaffolding помечен «после
   решения по стратегии обновления». Решение: да, целимся в copier, но
   строим его только когда сработают триггеры (4.4).

### 4.2 Конвенции v1, делающие стратегию рабочей

Нарушение любой из них ломает канал обновления, поэтому они попадают
в CLAUDE.md (Фаза 5) как жёсткие правила, а самые опасные нарушения
дополнительно ловятся CI (п.6).

1. **Проект создаётся только клоном, никогда копией файлов.** Процедура:
   `git clone` шаблона → `git remote rename origin template` →
   `git remote add origin <client-repo>`. Без общей git-истории merge
   превращается в свалку конфликтов (`--allow-unrelated-histories` —
   антипаттерн, в документации запрещён явно).
2. **Merge только тегов и только настоящим merge-коммитом.**
   `git merge --no-ff vX.Y.Z`; squash и rebase поверх влитых тегов
   запрещены — они уничтожают общего предка, и каждый следующий релиз
   конфликтует заново. Защита автоматическая, а не только конвенционная:
   проверка предка в CI (п.6а). `git rerere` включается bootstrap-скриптом
   (`make setup`) как локальное удобство — гарантией не считается (3.2).
3. **Шаблонные пути в клиентском проекте read-only: `core/`, `shared/`
   и фичи, приехавшие из шаблона в `modules/`.** Кастомизации — через
   предусмотренные архитектурой механизмы: собственные фичи-папки,
   конфигурация из окружения, события шины, шаблоны уведомлений, каталоги
   i18n. Если правки шаблонного пути не избежать: (а) обобщаемое — patch
   в шаблон и приезд обратно релизом; (б) действительно проектное —
   помечается комментарием `# CUSTOM(<project>): <причина>` и фиксируется
   в `CUSTOMIZATIONS.md` в корне проекта. Два специальных правила:
   изменение схемы таблиц шаблона запрещено даже как CUSTOM (только свои
   таблицы в ветке `project`, см. 3.5); фича, которую надо менять «сильно
   иначе», не правится по месту, а копируется под новым именем в проектный
   неймспейс — с записью в `CUSTOMIZATIONS.md`, что копия осознанно вышла
   из канала обновлений (оригинал отключается конфигом, физическое удаление —
   см. открытый вопрос №1). Всё непомеченное считается дрейфом.
4. **Релизная дисциплина шаблона.** SemVer: PATCH — фиксы, MINOR — новые
   возможности без ручной работы в клиентах, MAJOR — ломающие изменения
   публичных интерфейсов ядра или схемы БД. `CHANGELOG.md` в формате
   keep-a-changelog; у каждого MINOR/MAJOR — раздел «Upgrade notes»
   (что сделать руками); security-фиксы помечаются `[SECURITY]` и
   оформляются минимальными изолированными коммитами — это предусловие
   hotfix-процедуры (п.8). Константа `TEMPLATE_VERSION` обновляется релизом
   и логируется при старте приложения (наружу не публикуется — только логи
   и закрытая админ-диагностика).
5. **Alembic — мультиветочный** (branch labels по компонентам + ветка
   `project` у клиента, `depends_on` для межветочных зависимостей,
   накат строго `upgrade heads`), как в 3.5.
6. **CI-джоба `template-drift` в клиентском проекте.** Опорная точка —
   тег последнего влитого релиза; его имя джоба вычисляет из
   `TEMPLATE_VERSION` (`v${TEMPLATE_VERSION}`), а сам тег обязан находиться
   в origin проекта — для этого процедура обновления (п.8) содержит шаг
   `git push origin vX.Y.Z`; доступ к remote шаблона из CI не нужен.
   Три проверки:
   (а) **предок** — `git merge-base --is-ancestor "v${TEMPLATE_VERSION}"
   HEAD`; провал означает squash/rebase — блокирующая ошибка CI;
   (б) **дрейф** — `git diff "v${TEMPLATE_VERSION}" HEAD -- core/ shared/
   modules/` сверяется со списком в `CUSTOMIZATIONS.md`; незадекларированные
   отклонения — красный отчёт (report-only);
   (в) **владение Alembic-ветками** — правило из 3.5 (ревизии вне ветки
   `project` живут только в шаблонных путях и продолжают свою ветку).
7. **Реестр проектов.** Приватный файл (вне репозитория шаблона): проект →
   репозиторий → версия шаблона → дата обновления → дата deploy → прод-URL.
   Единственный источник ответа «фикс в проде» (3.7). Хостинг клиентских
   репозиториев — открытый вопрос №2.
8. **Процедура обновления — `docs/UPDATE.md`** (скелет в Фазе 1, финал
   в Фазе 5): fetch → merge тега → разрешение конфликтов по правилам →
   `git push origin vX.Y.Z` (тег в origin — для `template-drift`) →
   `uv sync` → `alembic upgrade heads` на dev-БД → `make lint && make test`
   → smoke (`/health`, `/ready`) → push → deploy → отметка в реестре.
   Два приложения: короткий чек-лист «раскатка security-фикса по всем
   проектам» и **hotfix-процедура для отставшего проекта**: если проект
   отстал настолько, что полный merge (с чтением Upgrade notes всех
   промежуточных релизов) не укладывается в SLA-окно, разрешается
   cherry-pick `[SECURITY]`-коммитов как документированное исключение —
   с отметкой в реестре («hotfix, catch-up pending») и обязательным
   полноценным merge вдогонку в течение 14 дней. Догоняющий merge обязан
   пройти Upgrade notes всех пропущенных релизов; git корректно сольёт
   уже применённый cherry-pick (общий предок сохранён — конвенция 2).

### 4.3 Что заложить в Фазах 1–6

| Фаза | Что добавляется под стратегию обновления |
|---|---|
| 1 | ADR «Стратегия обновления шаблона» (этот документ) и ADR «Мультиветочный Alembic» (включая раскладку миграций по компонентам — после решения открытого вопроса №6); `CHANGELOG.md` + тег `v0.1.0`; константа `TEMPLATE_VERSION` + лог при старте; каркас мультиветочного Alembic (branch labels, динамический `version_locations`); цель `make migrate` = `alembic upgrade heads` + CI-тест «после наката нет неприменённых голов»; `make setup` включает `rerere`; скелет `docs/UPDATE.md`; заготовка CI-джобы `template-drift` (три проверки из 4.2.6) |
| 2–4 | Миграции каждого модуля ядра — строго в своей Alembic-ветке; публичные интерфейсы ядра проектируются как контракт, ломать который можно только MAJOR-релизом |
| 5 | CLAUDE.md: конвенции 4.2 как правила и антипаттерны («не редактируй шаблонные пути без маркера», «не squash'ь и не rebase'ь merge шаблона», «не дописывай ревизии в чужие Alembic-ветки», «не меняй схему таблиц шаблона»); финальный `docs/UPDATE.md` + чек-лист security-раскатки + hotfix-процедура; **шаблон реестра проектов + read-only скрипт drift-check по реестру** (читает `TEMPLATE_VERSION` из main — отвечает «фикс влит»; «фикс в проде» — вручную по отметкам реестра); упоминание реестра в чек-листе продакшен-развёртывания |
| 6 | Приёмочная репетиция стратегии: создать «клиентский» проект клоном, кастомизировать (своя фича + одна помеченная правка ядра + одна незадекларированная — проверить, что `template-drift` её ловит), выпустить PATCH-релиз шаблона, пройти `docs/UPDATE.md` от начала до конца — процедура обязана сработать без правок самой процедуры. При проектировании `tools/add-feature` — решить открытый вопрос №5 (обновление фич, перенесённых не из шаблона) |

Ничего из перечисленного не является заготовкой под бэклог: это релизная
гигиена самого шаблона и документация процедуры. Уточнение границ: запрет
«скриптов массового обновления» касается только автоматизации **раскатки**;
read-only мониторинг (drift-check по реестру) разрешён и назначен Фазе 5.
Copier-код и генераторы в v1 не пишутся.

### 4.4 Путь эволюции

**Триггеры перехода на copier** (любой из):

- активных клиентских проектов ≥ 4–5;
- раскатка одного релиза регулярно занимает больше ~2 часов чистого
  времени (по оценкам 3.1 это наступает примерно там же, где и первый
  триггер);
- создание нового проекта требует повторяющейся ручной правки
  (переименования, вычистка модулей) — появилась потребность в scaffolding
  как таковом;
- два подряд релиза с болезненными конфликтами структуры (не контента).

**Как мигрировать** (оценка: 1–2 сессии на шаблон + одна штатная раскатка
релиза по проектам): в шаблон добавляется `copier.yml` (Jinja — только
в минимум файлов: имя проекта, `ENABLED_MODULES`, compose-имена); CI шаблона
получает тест генерации; выпускается первый copier-совместимый релиз.
Существующие проекты получают его **обычным git-merge** (это штатный
релизный цикл, не отдельный проект), и только после этого в каждом проекте
создаётся `.copier-answers.yml` с `_commit` = этот релиз и `_src_path`
шаблона — дальше проект живёт через `copier update`. Теги, выпущенные
до копиеризации, базой `_commit` служить не могут: `copier update` требует
`copier.yml` на базовой версии. Конвенции 4.2 переносятся без изменений;
`TEMPLATE_VERSION` и реестр остаются (реестр дополнительно читает
`_commit`). Это и есть момент, когда пункт бэклога «copier-scaffolding»
берётся в работу — решением Фазы 0 он разблокирован, но не начат.

**Судьба варианта Б.** Две независимые точки пересмотра: (1) ядро
стабилизировалось — после нескольких боевых проектов (ориентир: v2+)
за полгода не появилось ни одной правки внутри `core/` ни в одном проекте;
тогда вынос ядра в пакет даёт полностью автоматические обновления;
(2) масштаб приблизился к «десяткам» активных проектов — там даже
copier-раскатка пробивает бюджет ручного труда (§1 требование №3, §3.1),
и Б из опции становится целевым состоянием. До срабатывания любой из точек
вариант Б не рассматривается и абстракции под него не строятся.

## 5. Открытые вопросы

1. **Состав клиентского проекта: полный клон или урезанный?**
   Суть: держать ли в клиентском проекте все модули шаблона (неиспользуемые
   выключены через `ENABLED_MODULES`) или физически удалять лишние папки.
   Варианты: (а) полный клон, выключение конфигом; (б) удаление лишнего.
   Рекомендация: (а) для v1 — merge остаётся чистым (удалённые папки дают
   modify/delete-конфликты на каждом релизе); физическая вычистка станет
   бесплатной после перехода на copier (решается на этапе генерации).
   Что меняется от выбора: (б) сейчас = хронические конфликты при
   обновлениях и довод в пользу более раннего перехода на copier.

2. **Где хостятся клиентские репозитории?**
   Суть: доступ к репозиториям — предусловие раскатки фиксов. Дополнительное
   следствие конвенции 4.2.1: клиентский репозиторий несёт **полную
   git-историю шаблона** (весь код ядра и всех модулей, вся эволюция) —
   то есть каждому клиенту фактически передаётся полный исходник шаблона;
   это вопрос лицензии/договора независимо от места хостинга (связан с №4).
   Варианты: (а) собственная GitHub-организация разработчика, клиенту —
   deploy и/или mirror; (б) репозитории в аккаунтах клиентов с выданным
   доступом. Рекомендация: (а) — иначе security-SLA зависит от чужих
   админов, а массовые операции по реестру становятся невозможными при
   отзыве доступа; и при (а), и при (б) в договор нужен пункт о правах
   на код шаблона. Что меняется: при (б) в договоры и `docs/UPDATE.md`
   нужен пункт о гарантированном доступе, а реестр — про учёт внешних
   доступов.

3. **Политика поддержки версий шаблона.**
   Суть: обязаны ли проекты идти только вперёд, или security-фиксы
   бэкпортируются в старые ветки. Варианты: (а) только вперёд — все активные
   проекты обновляются до последнего релиза текущей MAJOR (для отставших
   действует hotfix-процедура 4.2.8); (б) release-ветки `1.x`, `2.x`
   с бэкпортом security. Рекомендация: (а) с единственным исключением —
   после выхода новой MAJOR прошлая MAJOR получает только
   `[SECURITY]`-патчи в течение согласованного окна (например, 6 месяцев).
   Что меняется: (б) в полном виде — постоянные бэкпорты, непосильные
   для соло; (а) без исключения — принуждение клиентов к платным
   MAJOR-апгрейдам ради security-фиксов, репутационный риск.

4. **SLA и договорная сторона обновлений.**
   Суть: обновление клиентского прод-проекта — работа; кто её оплачивает
   и в какой срок она обещана. Целевой SLA 24–48 ч из §1 вступает в силу
   как обязательство только после решения этого вопроса. Варианты:
   (а) поддержка по подписке (security — включено, feature-апгрейды —
   по запросу); (б) разовые работы по запросу клиента. Рекомендация: (а) —
   иначе security-SLA не имеет экономического основания, а необновляемые
   проекты станут репутационной миной; сюда же — договорный пункт о правах
   на код шаблона (см. №2). Что меняется: при (б) реестр обязан различать
   «поддерживаемые» и «замороженные» проекты, и для замороженных стратегия
   фактически не действует.

5. **Обновление фич, перенесённых через `tools/add-feature` не из шаблона.**
   Суть: в v1 все фичи живут в шаблоне и обновляются merge'ем вместе с
   ядром; но add-feature позволит переносить фичи между клиентскими
   проектами — такие копии выпадают из канала обновлений. Варианты:
   (а) принять: фича, скопированная из другого проекта, обновляется вручную
   по diff; при принятии этого варианта в Фазе 6 add-feature начнёт писать
   в переносимый `feature.toml` поле `origin_template_version` (с какого
   релиза скопирована фича — база для diff); до принятия решения схема
   манифеста не расширяется; (б) в будущем — «библиотека фич» как отдельные
   тегированные источники. Рекомендация: (а) для v1; вопрос вернуть
   в проектирование add-feature в Фазе 6. Что меняется: (б) — отдельная
   инфраструктура, явно за пределами v1, решение стоит принимать после
   первых реальных переносов.

6. **Раскладка миграций: по компонентам или корневым каталогом?**
   Суть: стратегия (§3.5) кладёт миграции каждого компонента ядра в его
   папку (`core/auth/migrations/`, ...), а корневой `migrations/` из
   структуры мастер-промпта вырождается в `env.py` + конфигурацию Alembic —
   это отклонение от зафиксированной структуры, и оно должно быть
   утверждено явно. Варианты: (а) миграции по компонентам, корневой
   `migrations/` — только env/конфиг; (б) все миграции в корневом
   `migrations/` с подпапками по веткам. Рекомендация: (а) — симметрично
   анатомии фич (у фич `migrations/` в папке уже зафиксирован
   мастер-промптом), миграции переезжают вместе с компонентом, и
   `tools/add-feature` не требует специальной обработки. Что меняется:
   (б) сохраняет букву структуры, но ломает симметрию с фичами, усложняет
   перенос фич и повышает вероятность конфликтов в общем каталоге.
   Решение фиксируется в ADR «Мультиветочный Alembic» (Фаза 1).
   **Решение (2026-07-06): (а) — миграции по компонентам, корневой
   `migrations/` — только env/конфиг (ОВ-10).**
