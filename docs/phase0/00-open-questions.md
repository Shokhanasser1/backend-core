## Блок 1. Блокируют утверждение схемы / старт Фаз 1–2

Вопросы, влияющие на таблицы, публичные интерфейсы или структуру
репозитория. Без решения по ним схема БД и карта интерфейсов не могут
считаться утверждёнными, а Фазы 1–2 — стартовавшими без риска переделки.

### Тема: auth / tenants

#### ОВ-01. Глобальные `users` vs per-tenant users
- **Суть:** схема принимает глобальные `users` (и вслед — `user_totp`,
  `user_recovery_codes`, `refresh_tokens`); это сознательное отступление от
  буквы мастер-промпта «tenant_id во всех бизнес-таблицах». Сюда же —
  отсутствие RLS по `app.user_id` на auth-таблицах в v1 (защита на уровне
  сервиса auth, схема §3.5).
- **Варианты:** (а) глобальные users + memberships — один вход, несколько
  организаций, одна 2FA; (б) per-tenant users — буквальное соответствие
  правилу, но дубли личности, повторный вход при смене организации,
  N секретов 2FA.
- **Рекомендация:** (а); требуется явное утверждение владельцем, поскольку
  это отступление от зафиксированной формулировки мастер-промпта.
- **Источник:** 01 №9.
- **Блокирует:** утверждение схемы; Фаза 2.
- **Статус:** ✅ (а) глобальные `users` + memberships — утверждено владельцем 2026-07-06.

#### ОВ-02. Удаление пользователя / право на забвение — рекомендации документов ПРОТИВОРЕЧАТ
- **Суть:** субъект ПДн вправе требовать уничтожения данных (ЗРУ-547), но
  append-only аудит и платёжная история должны сохраняться. Документы 01 и
  03 предлагают разные механики и их рекомендации противоречат — нужен
  единый выбор.
- **Позиция схемы (01 №6):** жёсткое удаление user (разрешено, если не
  владеет тенантами) + конвенция «в `audit_log.payload` и
  `notification_outbox.params` нет ПД, только идентификаторы» + чистка по
  совпадению при удалении (`notification_outbox.recipient`,
  `invitations.email`) + ретенция терминальных строк outbox/invitations
  N дней (default 90, env); `audit_log` не трогается (хранит только UUID).
  Псевдонимизацию журнала 01 отвергает как дорогую и подрывающую
  append-only.
- **Позиция threat model (03 №3):** анонимизация — запись `users` затирается
  на месте, `user_id` в аудите остаётся «мёртвым» UUID; полное каскадное
  удаление названо вариантом B и не рекомендовано.
- **Варианты (сведённые):** (а) жёсткое удаление + чистка ПД за пределами
  каскада + ретенция (позиция 01); (б) анонимизация записи `users` на месте
  (позиция 03). В обоих случаях юридическую достаточность для ЗРУ-547
  проверить юристом (связка с ОВ-31).
- **Рекомендация:** единой нет — требуется выбор владельца; выбранный
  вариант согласовать в обоих документах и реализовать в Фазе 2 (03 №3:
  «меняется схема auth и один эндпоинт»).
- **Источник:** 01 №6; 03 №3.
- **Блокирует:** утверждение схемы (механика удаления §2.1); Фаза 2.
- **Статус:** ✅ (а) жёсткое удаление + чистка ПД за пределами каскада + ретенция
  (позиция 01) — утверждено владельцем 2026-07-06. Рекомендация 03 (анонимизация)
  отклонена, документ 03 приведён в соответствие; юридическая достаточность
  проверяется юристом в пакете ОВ-31.

#### ОВ-03. Как тенант попадает в `TenantContext`: клейм JWT vs заголовок
- **Суть:** механизм передачи tenant-контекста в API-запросе определяет
  дизайн JWT, refresh-флоу и стоимость проверки прав на каждый запрос.
- **Варианты:** (а) tenant-клейм в JWT — access-токен на пару user×tenant,
  смена тенанта = обмен через refresh; контекст подписан,
  `require_permission` не ходит в БД за membership; (б) заголовок
  `X-Tenant-Id` + проверка membership на каждом запросе — один токен на все
  тенанты, но контекст подделываем заголовком и проверка дороже.
- **Рекомендация:** (а); `list_user_tenants` обслуживает экран выбора
  тенанта до выдачи tenant-токена.
- **Источник:** 02 №9.
- **Блокирует:** утверждение интерфейсов; дизайн JWT/refresh — Фаза 2.
- **Статус:** ✅ (а) tenant-клейм в JWT — утверждено владельцем 2026-07-06.

#### ОВ-04. Носитель платформенной роли: флаг вне тенантов vs служебный тенант
- **Суть:** admin-каркас обслуживает tenant- и platform-уровни одним
  механизмом прав, но нужен носитель платформенной роли (управление
  тенантами, `TenantService.set_status`, просмотр всех платежей). Схема уже
  содержит `users.is_superuser`; на этот вопрос ссылаются V1/V10 threat
  model (аудируемый кросс-тенантный путь платформенного админа).
- **Варианты:** (а) флаг/роль `platform_admin` вне тенантов; (б) служебный
  «платформенный тенант».
- **Рекомендация:** (а) — проще в RLS и проверке прав.
- **Источник:** 02 №7 (используется в 03 V1, V10).
- **Блокирует:** схему auth и допустимость `ctx.tenant_id is None`
  (интерфейсы §2.1); Фазы 2 и 4.
- **Статус:** ✅ (а) флаг/роль `platform_admin` вне тенантов — утверждено владельцем 2026-07-06.

### Тема: billing

#### ОВ-05. Merchant-креды платёжек: на деплой или per-tenant
- **Суть:** один вопрос в двух документах (рекомендации совпадают). Биллинг
  ядра v1 — подписки тенантов в пользу владельца развёртывания, поэтому
  естественный дефолт — один merchant-аккаунт Payme/Click на инсталляцию.
- **Варианты:** (а) один набор кредов на клиентский проект, только env —
  типовой случай «шаблон = один бизнес»; (б) креды per-tenant в БД
  (маркетплейс-сценарий) — шифрованное хранение по паттерну
  `notification_settings.config_encrypted`, write-only-операции, ротация,
  выбор ключа проверки вебхука по платежу.
- **Рекомендация:** (а) для v1; per-tenant — в бэклог, отдельным ADR при
  реальном спросе клиента.
- **Источник:** 02 №3; 03 №10.
- **Блокирует:** утверждение схемы billing (при (б) — новые
  таблицы/колонки) и конфиг адаптеров; Фаза 3.
- **Статус:** ✅ (а) один набор кредов на деплой, только env — утверждено владельцем 2026-07-06.

#### ОВ-06. Одна живая подписка на тенанта
- **Суть:** ограничение зафиксировано partial-unique-индексом
  `uq_subscriptions_one_live_per_tenant`. Если возможен сценарий «основной
  план + параллельные платные дополнения» — индекс надо снимать и вводить
  тип подписки.
- **Варианты:** (а) оставить ограничение — дополнения трактуются как
  лимиты/фичи тарифов, т.е. модуль saas из бэклога; (б) снять индекс и
  ввести тип подписки.
- **Рекомендация:** (а).
- **Источник:** 01 №7.
- **Блокирует:** утверждение схемы (индекс `subscriptions`).
- **Статус:** ✅ (а) ограничение остаётся — утверждено владельцем 2026-07-06.

### Тема: архитектура и сквозные конвенции

#### ОВ-07. Result vs типизированные исключения — расхождение с мастер-промптом
- **Суть:** мастер-промпт перечисляет `Result` в составе shared/; документ
  интерфейсов фиксирует иерархию исключений `DomainError` (единый маппинг
  на HTTP/i18n в одном хендлере).
- **Варианты:** (а) типизированные исключения — один маппинг на HTTP/i18n,
  нет `unwrap`-шума; (б) Result — явные ошибки в сигнатурах ценой шума без
  checked-типов.
- **Рекомендация:** (а); упоминание Result в мастер-промпте считать
  устаревшей записью. Требует утверждения владельцем — отступление от
  источника правды.
- **Источник:** 02 №8.
- **Блокирует:** Фаза 1 (структура shared/); утверждение интерфейсов.
- **Статус:** ✅ (а) типизированные исключения `DomainError` — утверждено владельцем
  2026-07-06; упоминание Result в мастер-промпте помечено устаревшим (правка внесена).

#### ОВ-08. Префикс модуля в именах таблиц фич
- **Суть:** пример в мастер-промпте — `owns_tables = ["carts",
  "cart_items"]` (без префикса). Выбор влияет на `feature.toml` всех будущих
  фич — менять потом дорого.
- **Варианты:** (а) префикс модуля (`commerce_carts`, `commerce_cart_items`)
  — исключает коллизии имён между будущими модулями (crm и commerce оба
  захотят `orders`/`contacts`), мгновенно показывает владельца таблицы в
  psql; (б) без префикса — буква мастер-промпта.
- **Рекомендация:** (а).
- **Источник:** 01 №1.
- **Блокирует:** конвенцию §4 схемы (фиксируется при утверждении); жёсткий
  дедлайн — до Фазы 6.
- **Статус:** ✅ (а) префикс модуля (`commerce_carts`) — утверждено владельцем 2026-07-06;
  пример в мастер-промпте исправлен.

#### ОВ-09. Имена событий шины: с префиксом модуля или короткие
- **Суть:** мастер-промпт использует `order.created` / `cart.checked_out`;
  документ интерфейсов фиксирует `commerce.order.created`. От выбора зависит
  содержимое `publishes_events`/`listens_events` во всех манифестах.
- **Варианты:** (а) префиксованные — нет коллизий между будущими модулями
  (crm/commerce), грепаемость, 1:1 с `action` в audit; (б) короткие — как в
  мастер-промпте, лаконичнее.
- **Рекомендация:** (а); примеры в мастер-промпте и feature.toml считать
  устаревшей записью.
- **Источник:** 02 №1.
- **Блокирует:** утверждение карты событий (интерфейсы §6); реестр шины —
  Фаза 1; манифесты — Фаза 6.
- **Статус:** ✅ (а) имена с префиксом модуля (`commerce.order.created`) — утверждено
  владельцем 2026-07-06; примеры в мастер-промпте исправлены.

### Тема: стратегия обновления / структура репозитория

#### ОВ-10. Раскладка миграций: по компонентам или корневым каталогом
- **Суть:** стратегия обновления (§3.5) кладёт миграции каждого компонента
  ядра в его папку (`core/auth/migrations/`, ...), корневой `migrations/`
  вырождается в `env.py` + конфигурацию Alembic — отклонение от структуры
  мастер-промпта, требует явного утверждения.
- **Варианты:** (а) миграции по компонентам, корневой каталог — только
  env/конфиг; (б) все миграции в корневом `migrations/` с подпапками по
  веткам.
- **Рекомендация:** (а) — симметрично анатомии фич, миграции переезжают
  вместе с компонентом, `tools/add-feature` не требует спецобработки.
  Фиксируется в ADR «Мультиветочный Alembic» (Фаза 1).
- **Источник:** 04 №6.
- **Блокирует:** Фаза 1 (каркас Alembic, структура репозитория).
- **Статус:** ✅ (а) миграции по компонентам, корневой каталог — только env/конфиг —
  утверждено владельцем 2026-07-06; зафиксировано в ADR «Мультиветочный Alembic».

### Тема: auth (к Фазе 2)

#### ОВ-11. Параметры безопасности auth — значения по умолчанию
- **Суть:** предложены дефолты шаблона: access JWT 10 мин, refresh 30 дней,
  lockout после 5 неудач с экспоненциальной задержкой до 15 мин, без
  jti-denylist для access в v1 (компрометация access живёт максимум его TTL).
- **Варианты:** (а) принять предложенные цифры как дефолты шаблона;
  (б) скорректировать значения.
- **Рекомендация:** (а) — подтвердить цифры.
- **Источник:** 01 №5.
- **Блокирует:** Фаза 2 (конфиг auth).
- **Статус:** ⬜

#### ОВ-12. Политика ротации refresh при легитимных ретраях клиента
- **Суть:** гонка ротации закрыта атомарным CAS, но сетевой ретрай клиента
  (мобильные сети, таймаут после успешной ротации) предъявит уже
  ротированный токен и по строгой политике убьёт всё семейство (повторный
  вход).
- **Варианты:** (а) строгая — любой повтор = reuse → отзыв семейства +
  audit-событие; (б) grace-период 30–60 с — повтор только что ротированного
  токена возвращает того же потомка (требует ссылку на потомка или кэш в
  Redis).
- **Рекомендация:** (а) для v1 — проще и безопаснее, поведение видно в
  audit; вернуться к (б), если пилот покажет заметные ложные срабатывания.
- **Источник:** 01 №10.
- **Блокирует:** Фаза 2.
- **Статус:** ⬜

#### ОВ-13. Сброс пароля — в scope Фазы 2?
- **Суть:** в мастер-промпте для Фазы 2 не назван (регистрация, вход,
  refresh, 2FA, RBAC), но без него шаблон в прод не выйдет; интерфейс уже
  включён в карту (02 §3.1).
- **Варианты:** (а) включить в Фазу 2; (б) исключить — убрать 2 метода и
  событие `auth.user.password_reset_requested`.
- **Рекомендация:** (а).
- **Источник:** 02 №4.
- **Блокирует:** Фаза 2 (состав).
- **Статус:** ⬜

#### ОВ-14. Recovery-коды для 2FA
- **Суть:** в карте интерфейсов только TOTP; потеря устройства = потеря
  аккаунта (решается вручную через админа). Схема БД уже содержит
  `user_recovery_codes`. Смежен с ОВ-15, но это отдельное решение.
- **Варианты:** (а) добавить одноразовые recovery-коды в Фазу 2 — малая
  цена, стандарт безопасности; (б) без них + задокументировать процедуру
  ручного сброса 2FA админом.
- **Рекомендация:** (а).
- **Источник:** 02 №5.
- **Блокирует:** Фаза 2 (состав).
- **Статус:** ⬜

#### ОВ-15. Обязательная 2FA для админов
- **Суть:** принуждать ли platform-admin и tenant-owner к TOTP (нельзя
  войти без настройки 2FA). Смежен с ОВ-14 (recovery-коды), но решается
  отдельно.
- **Варианты:** (а) да для platform-admin; для tenant-owner — настройка
  тенанта (по умолчанию выключено); (б) принуждать обоих; (в) никого не
  принуждать.
- **Рекомендация:** (а) — одна проверка в auth-флоу Фазы 2.
- **Источник:** 03 №4.
- **Блокирует:** Фаза 2.
- **Статус:** ⬜

#### ОВ-16. Анти-стаффинг сверх rate limit
- **Суть:** распределённый low-and-slow credential stuffing с тысяч IP
  лимитами по IP не ловится полностью; CAPTCHA/Proof-of-Work и проверка
  паролей по базам утечек — в v1 или бэклог?
- **Варианты:** (а) бэклог — в v1 достаточно rate limit (включая

#### ОВ-39. Модель доступа покупателя (storefront, commerce)
- **Суть:** покупатель (`customer_user_id`) — аутентифицированный
  пользователь, но не член тенанта: тенантный RBAC для него неприменим, а
  `public_endpoint` — ложь (корзина и заказы требуют входа).
- **Варианты:** (а) авто-роль `customer` в тенанте при первом заказе —
  единый RBAC, но membership раздувается B2C-объёмами;
  (б) `authenticated_endpoint` + ownership-проверка в сервисе (объект
  принадлежит `ctx.actor`) — маркер уже введён; (в) отдельная ветка
  customer-прав вне membership.
- **Рекомендация:** (б) — не раздувает membership, вторая линия остаётся в
  сервисе; детальный дизайн — Фаза 6, но решение нужно до схемы commerce.
- **Источник:** 02 №10.
- **Блокирует:** Фаза 6 (до проектирования схемы commerce).
