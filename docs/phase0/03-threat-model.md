## 1. Активы и границы доверия

### 1.1. Что защищаем

Защищаемые **операции**: вход/refresh, смена пароля и 2FA, выдача и отзыв прав, приём платёжных вебхуков, списание/активация подписки, рассылка уведомлений, действия платформенного админа.

### 1.2. Границы доверия

1. **Интернет ↔ API** — всё извне недоверенно, включая вебхуки платёжек (их аутентичность доказывается подписью, а не источником; привязка вебхука к тенанту выводится только из наших собственных записей, не из данных запроса — см. V4).
2. **Тенант ↔ тенант** — внутренняя граница внутри одной БД; держится на двух независимых механизмах (автофильтрация репозитория + RLS).
3. **Пользователь ↔ админ** — раздельные плоскости прав (RBAC), tenant-админ ≠ platform-админ.
4. **Приложение ↔ внешние сервисы** — Payme, Click, Eskiz, Telegram, SMTP, Sentry — третья сторона: наружу уходит минимум данных, внутрь ничего не принимается без валидации/подписи.
5. **Код ↔ CI/CD ↔ зависимости** — граница цепочки поставки.

## 2. Топ-10 векторов атаки

Состав совпадает с заданным минимумом — по анализу активов он и есть
топ-10 для мультитенантного ядра, обрабатывающего деньги и ПДн; замен
не требуется.

Рассмотрено и **исключено из топ-10: SSRF**. В v1 нет исходящих
запросов по URL, которыми управляет пользователь: адаптеры платёжек,
SMS и Telegram ходят на фиксированные адреса из конфига, загрузки
файлов/аватарок в scope v1 отсутствуют. При появлении `core/files` и
аватарок из бэклога вектор оживает — тогда обязательны allowlist схем
и хостов и запрет исходящих запросов в приватные диапазоны адресов.

### V1. Cross-tenant утечка данных (IDOR/BOLA)

**Кто и как.** Легитимный пользователь тенанта A (или атакующий с любым дешёвым аккаунтом) перебирает идентификаторы объектов (`GET /orgs/{id}`, id инвайта, id платежа) и читает/меняет данные тенанта B. Самый вероятный и самый дорогой инцидент для мультитенантного SaaS.

**Точка входа.** Любой эндпоинт, принимающий id объекта.

**Механизмы защиты.**
- Колонка `tenant_id` во всех бизнес-таблицах; **автофильтрация в репозитории**: базовый Repository всегда добавляет `WHERE tenant_id = :current_tenant` из контекста запроса — «забыть фильтр» на уровне фичи невозможно, фильтра там просто нет.
- **RLS в Postgres** как независимая вторая линия: политики `USING (tenant_id = app_current_tenant_id())` на каждой бизнес-таблице. Контекст тенанта ставится **транзакционно-локально**: `SELECT set_config('app.tenant_id', :tenant_id, true)` (семантика `SET LOCAL`) в начале каждой транзакции и автоматически очищается на commit/rollback — переиспользование соединений пулом asyncpg **не может** протащить контекст тенанта A в транзакцию тенанта B; сессионный `SET` запрещён конвенцией и ревью. Хелпер `app_current_tenant_id()` нормализует пустой GUC в NULL → отсутствие контекста означает **fail closed**: ноль строк, а не все строки и не ошибка. Механика, роли и подводный камень с reset-значением GUC зафиксированы в схеме §3.1–3.2.
- Даже сырой запрос мимо репозитория чужого тенанта не увидит: runtime-роли не владеют таблицами и не имеют `BYPASSRLS`.
- UUID вместо последовательных id — перебор не даёт валидных ссылок.
- Чужой объект отдаётся как **404, не 403** — не подтверждаем существование.
- Обязательный тест изоляции тенантов на каждую новую таблицу (Definition of Done) + обязательные тесты механики контекста: транзакция без контекста возвращает 0 строк; два последовательных запроса разных тенантов через одно физическое соединение пула не наследуют контекст друг друга (§4).

**Остаточный риск.** Логические обходы: пути, легитимно работающие поверх тенантов. Их три класса, все — явные и тестируемые: (а) **системные операции вне tenant-контекста** — исчерпывающий список через отдельную роль `app_maintenance` (схема §3.4: регистрация организации, сопоставление вебхука с платежом — см. V4, диспетчер outbox, ретенционные свипы); каждая новая такая операция — событие уровня ревью, не рутина; (б) **платформенный админ** — не обход RLS, а явное право с аудитом каждого использования (см. V10; носитель платформенной роли — открытый вопрос №7 документа интерфейсов); (в) **фоновые задачи arq** — конвенция «задача получает `tenant_id` в аргументах и работает как `app_user` в его контексте» закладывается в базовые классы shared/ в Фазе 1 и обязательна с первого реального использования очереди (Фаза 3, notifications). Join'ы к глобальным справочникам (`currencies`, `plans`) безопасны by design — справочники не содержат тенантных данных и для `app_user` read-only.

### V2. Перебор паролей / credential stuffing

**Кто и как.** Внешний атакующий: онлайн-брутфорс одного аккаунта или прогон утёкших пар email/пароль по `/auth/login` с ботнета; отдельно — брутфорс 6-значного TOTP на шаге 2FA (пространство всего 10^6 кодов).

**Точка входа.** `/auth/login`, `/auth/password-reset`, `/auth/2fa/verify`.

**Механизмы защиты.**
- **Rate limiting в Redis**: лимиты по ключам `ip`, `ip+login` **и `login`** — per-account счётчик неудачных попыток, работающий независимо от IP (`auth:fail:{user_id}` / `auth:lock:{user_id}` по схеме §2.1); экспоненциальные задержки, временная блокировка при переборе. Без per-account ключа распределённый перебор одного аккаунта с многих IP не ловился бы вовсе.
- **Лимит попыток на `/auth/2fa/verify`** — отдельный счётчик по ключу `user` (не только IP): N неверных кодов подряд → временный локаут шага 2FA + событие безопасности и уведомление пользователю. Без этого перебор TOTP в 30-секундных окнах реален.
- **argon2id** — офлайн-перебор украденного дампа хэшей дорог по памяти и времени.
- 2FA (TOTP) — украденный пароль сам по себе не даёт входа.
- Единообразный ответ «invalid credentials» и выровненное время ответа (проверка dummy-хэша при несуществующем логине) — нет user enumeration на `/auth/login`.
- **Password reset без enumeration**: ответ одинаков для существующего и несуществующего email («если такой аккаунт есть — письмо отправлено»), время выровнено. Reset-токен: одноразовый, короткий TTL, хранится только хэшем (Redis с TTL ~1 ч — схема §2.1); успешная смена пароля инвалидирует все выданные reset-токены и все refresh-сессии.
- Аудит неуспешных входов (ip, user_agent, request_id) — материал для реагирования (механика журнала до Фазы 4 — открытый вопрос №11).
- Пороговые уведомления пользователю о серии неудачных входов — через notifications.

**Остаточный риск.** Распределённый low-and-slow стаффинг с тысяч IP лимитами по IP не ловится полностью. CAPTCHA/Proof-of-Work и проверка паролей по базам утечек — вне v1 (см. Открытые вопросы, №5). Смягчение — 2FA и per-account лимит `login` независимо от IP.

### V3. Кража, подделка или повтор токенов

**Кто и как.** Атакующий, получивший refresh-токен (XSS у клиента, вредонос на устройстве, утечка из логов/бэкапа клиентского приложения), бесконечно продлевает себе сессию жертвы. Либо форжит access JWT, обходя верификацию подписи манипуляцией заголовком `alg`.

**Точка входа.** `/auth/refresh`; любой эндпоинт, принимающий access JWT.

**Механизмы защиты.**
- Короткоживущий access JWT (минуты) + **ротация refresh при каждом обновлении**: старый токен немедленно инвалидируется, выдаётся новый.
- **Детект повторного использования**: приход уже ротированного refresh = маркер кражи → отзыв всей цепочки (token family), принудительный разлогин, событие в аудит и уведомление пользователю. Гонка конкурентной ротации исключена атомарным CAS на уровне БД (схема §2.1).
- **Строгая верификация алгоритма подписи**: JWT верифицируется только алгоритмом, заданным в конфигурации (allowlist из одного значения); заголовок `alg` из самого токена никогда не выбирает алгоритм. `alg: none` и любой неожидаемый алгоритм → отказ. Это закрывает класс alg-confusion: подстановку `none` сейчас и верификацию RS256-токена как HS256 с публичным ключом в роли HMAC-секрета — при будущей миграции на асимметрию (см. открытый вопрос №8).
- В БД хранится только **хэш** refresh-токена — дамп БД не даёт живых токенов.
- Отзыв: logout, смена пароля и отключение 2FA убивают все refresh пользователя.
- Токены никогда не пишутся в логи (denylist-фильтр логгера) и не попадают в Sentry (scrubbing).

**Остаточный риск.** Окно жизни access-токена: украденный access работает до истечения (минуты) — отзыв мгновенно его не убивает; это осознанный трейдофф stateless-JWT. Компрометация ключа подписи JWT = катастрофа → ключ только из env, ротация ключа описывается в Фазе 5 (deployment checklist). Транспорт refresh-токена на стороне клиента (тело ответа vs httpOnly-cookie) — открытый вопрос №14; в рекомендованном для v1 варианте (тело ответа) cookie не используется, поэтому CSRF-поверхности на `/auth/refresh` нет; при введении cookie-режима CSRF-защита (`SameSite` + анти-CSRF-токен) обязательна.

### V4. Подделка, повторная и «чужая» доставка платёжных вебхуков

**Кто и как.** Атакующий шлёт на `/billing/webhooks/payme|click` сфабрикованный колбэк «платёж успешен» и получает подписку бесплатно; честный вебхук доставляется повторно (ретраи провайдера или намеренный replay) и создаёт дубль начисления; два одинаковых вебхука приходят **одновременно** и оба проходят проверку состояния; наконец, вебхук пытаются провести по платежу чужого тенанта.

**Точка входа.** Публичные webhook-эндпоинты (аутентификации пользователя у них нет by design; tenant-контекста из сессии — тоже нет).

**Механизмы защиты.**
- **Проверка аутентичности по протоколу провайдера** в каждом адаптере: Payme — секрет в Basic-auth заголовке Merchant API; Click — контрольная строка `sign_string` с секретным ключом. Сравнение — константное по времени. Неверная подпись → отказ по протоколу провайдера + запись в журнал (`payment_webhooks`, схема §2.3).
- **Привязка вебхука к тенанту — только через наши записи.** Тенант никогда не берётся из данных запроса: после проверки аутентичности адаптер находит платёж в нашей БД по `(provider, provider_transaction_id)` либо по нашему `payment_id` (он же merchant-идентификатор — схема §2.3) — это узкая, явно помеченная системная операция вне tenant-контекста (точечный SELECT через `app_maintenance`/`SystemRepository`); из найденного платежа берётся его `tenant_id`, транзакция обработчика получает `app.tenant_id`, и вся дальнейшая обработка выполняется в tenant-контексте под RLS. Вебхук по неизвестному платежу отвергается и фиксируется как аномалия. Следствие: подтверждённый вебхук по платежу тенанта A ни при каких данных запроса не может изменить объекты тенанта B.
- **Идемпотентность**: ключ дедупликации `(provider, dedup_key)` с уникальным индексом в журнале `payment_webhooks` (`dedup_key` адаптер вычисляет из id транзакции и метода/действия — схема §2.3) — провайдер обязан входить в ключ, пространства id Payme и Click независимы и могут коллидировать; повтор — тот же ответ без побочных эффектов.
- **Сериализация конкурентных доставок**: INSERT строки журнала вебхуков и переход состояния платежа — одна транзакция; конкурентный дубль ловит конфликт по `(provider, dedup_key)` и блокируется на строке первого вебхука (`SELECT ... FOR UPDATE`) до её коммита, после чего отвечает идемпотентно из уже совершённого перехода (механика — схема §2.3); эффект (активация подписки, начисление) — ровно один.
- **Сверка суммы и валюты** с ожидаемым платежом до перехода состояния. Внутри системы суммы — только целые в минимальных единицах + экспонента из справочника (UZS=0, USD=2): float-арифметики в деньгах нет вообще. **Но на границе адаптера конверсия единиц провайдера остаётся**: Payme Merchant API считает суммы в тийинах (сум × 100), Click присылает сумму в сумах с дробной частью — класс ошибок «×100» и float-парсинга живёт именно здесь, и внутренний тип данных от него не защищает. Конверсия локализована в одной функции каждого адаптера и покрыта обязательным тестом (§4).
- **Машина состояний платежа**: допустимые переходы фиксированы (`created → pending → succeeded | failed | canceled | expired` — интерфейсы §4.1); «оплата» несуществующего или уже оплаченного платежа отвергается.
- Полный след каждого вебхука (сырой payload с отредактированными секретными заголовками, вердикт подписи) — журнал `payment_webhooks` ведётся с Фазы 3; связка с общим аудитом — открытый вопрос №11.

**Остаточный риск.** Утечка merchant-секрета у нас или у провайдера делает подпись бесполезной — смягчение: секреты в env, аудит аномалий (вебхуки по неизвестным платежам), возможность быстрой ротации ключа. Ошибка конверсии единиц на границе адаптера ловится только тестом и ревью адаптера — поэтому тест конверсии обязателен на каждый адаптер. Расхождения из-за сбоя провайдера ловятся только сверкой-reconciliation (v1 — ручная процедура, см. Открытые вопросы, №7). Модель «один мерчант на развёртывание vs per-tenant» — открытый вопрос №10; при per-tenant выбор ключа проверки подписи должен следовать той же привязке «платёж → тенант → его ключ».

### V5. SQL- и командные инъекции

**Кто и как.** Внешний атакующий через любые входные поля; классика, но при деньгах в БД цена ошибки максимальна.

**Точка входа.** Любой параметр API, заголовки, payload вебхуков.

**Механизмы защиты.**
- **Только параметризованные запросы через SQLAlchemy ORM/Core**; конкатенация SQL запрещена стандартом; редкий `text()` — только с bound-параметрами, ловится на ревью и **bandit** в CI.
- **Pydantic-схемы на границе API** — ни один невалидированный байт внутрь не проходит; типы, длины, enum'ы, `extra="forbid"`.
- Командных инъекций негде взяться: приложение не вызывает shell; появление `subprocess`/`os.system` — сигнал bandit.
- Приложение работает в БД непривилегированной ролью + RLS — даже успешная инъекция в одном запросе ограничена данными текущего тенанта и правами роли.

**Остаточный риск.** Инъекции второго порядка через данные, попавшие в фоновые задачи/шаблоны уведомлений (Jinja-шаблоны рендерятся с autoescape и без доступа к контексту приложения); сырой SQL в миграциях Alembic — дисциплина ревью.

### V6. Утечка секретов через логи, репозиторий, Sentry

**Кто и как.** Не атака, а самострел с последствиями атаки: токен в трейсбеке, merchant-ключ в закоммиченном `.env`, пароль в теле запроса, улетевшем в Sentry. Репозиторий-шаблон клонируется в десятки клиентских проектов — утечка тиражируется.

**Точка входа.** git-история, JSON-логи, события Sentry, вывод CI.

**Механизмы защиты.**
- Платформенные секреты **только из переменных окружения**; в репозитории — `.env.example` с фиктивными значениями; `.env` в `.gitignore` с Фазы 1. Per-tenant конфиги каналов — в БД только шифртекстом (A6): дамп БД и логи запросов не содержат открытых токенов.
- **JSON-логи со структурной фильтрацией**: denylist-процессор вырезает `password`, `token`, `authorization`, `secret`, коды 2FA, полные номера карт до записи; стандарт «логи никогда не содержат секретов» + тест на фильтр.
- **Sentry**: `send_default_pii=False`, `include_local_variables=False` (стектрейсы уходят без локальных переменных функций — denylist по именам полей их бы не поймал), `before_send`-скраббер (см. §3.4), тела запросов не прикладываются.
- CI: **bandit** (хардкод секретов в коде), поиск по паттернам ключей в самопроверке фазы; секрет-сканер gitleaks — кандидат в пайплайн (см. Открытые вопросы, №6).
- Секреты не проходят через шину событий и аргументы arq-задач (конвенция + ревью).

**Остаточный риск.** Секрет, попавший в git-историю до обнаружения, считается скомпрометированным навсегда — процедура: немедленная ротация, не «удаление коммита». Логи сторонних библиотек (например, HTTP-клиент на DEBUG) могут печатать заголовки — глобальный уровень для библиотек WARNING+. Скраббинг Sentry — best-effort: фильтр доказуемо чистит известные поля, но не может гарантировать отсутствие ПДн/секретов в произвольных строках сообщений — поэтому отключение locals обязательно, а не опционально.

### V7. Злоупотребление правами и эскалация (BFLA, mass assignment)

**Кто и как.** Обычный пользователь дёргает админский эндпоинт напрямую (BFLA); или передаёт в `PATCH /me` лишнее поле `role: "owner"` / `tenant_id` и повышает себя (mass assignment).

**Точка входа.** Любые mutating-эндпоинты; админ-роуты.

**Механизмы защиты.**
- **«Эндпоинт без декларации прав не поднимается»**: обязательная зависимость `require_permission(...)`; на старте приложение обходит все роуты и падает, если у роута нет маркера прав; на это есть тест. Забытая проверка = не работающее приложение, а не дыра.
- **Права проверяются на уровне сервиса**, не только роутера — прямой вызов сервиса из другой фичи или из задачи arq не обходит RBAC.
- **Mass assignment**: раздельные Pydantic-схемы create/update/read; в схемах записи нет системных полей (`id`, `tenant_id`, `role`, `is_active`, балансы); `extra="forbid"` — неизвестное поле = 422, а не молчаливое игнорирование.
- Раздельные плоскости: tenant-RBAC и platform-admin — разные permission-неймспейсы; выдача прав — сама по себе привилегированная операция, попадающая в аудит. Гранты системных ролей для тенантного админа неизменяемы на уровне RLS-политик (схема §3.3) — эскалация через правку `role_permissions` системной роли невозможна.
- Негативный тест на 403 — обязателен для каждого эндпоинта (Definition of Done).

**Остаточный риск.** Ошибки в самой карте «роль → permission» (слишком широкая роль) — механикой не ловятся, только ревью матрицы прав; матрица фиксируется в Фазе 2 и проверяется тестами на каждую роль.

### V8. Supply chain: зависимости и базовые образы

**Кто и как.** Атакующий компрометирует PyPI-пакет, его мейнтейнера или базовый Docker-образ; вредоносный код исполняется с полными правами приложения — доступ к БД, секретам, платёжным ключам. Для шаблона риск умножается: заражённое ядро разъезжается по всем клиентским проектам.

**Точка входа.** `pyproject.toml` / `uv.lock`, Dockerfile, GitHub Actions.

**Механизмы защиты.**
- Политика мастер-промпта: **только зрелые популярные библиотеки**, каждая новая зависимость требует обоснования — минимальная поверхность.
- **uv с lock-файлом** — воспроизводимые сборки, версии пиннятся, «тихий» апгрейд невозможен.
- CI-конвейер: **pip-audit** (известные CVE зависимостей) + **bandit** (SAST) + **trivy** (скан образа и базового слоя). Красный CI = не мержим.
- Минимальный базовый образ (slim), многостадийная сборка, контейнер без root.
- GitHub Actions пиннятся по версии; секреты CI — только через GitHub Secrets.

**Остаточный риск.** Zero-day в популярном пакете и malicious release до появления advisory сканеры не видят. Смягчение: lock-файл (новая версия не подтянется сама), осознанные регулярные обновления (процедура — Фаза 5), стратегия обновления шаблона доносит патч до клиентских проектов быстро.

### V9. DoS и исчерпание ресурсов

**Кто и как.** Конкурент/вымогатель/скрипт: флуд запросами, тяжёлые выборки без пагинации, гигантские тела запросов, забивание очереди arq (например, спам эндпоинтом, порождающим SMS — это ещё и прямые деньги за Eskiz), исчерпание пула соединений БД зависшими внешними вызовами.

**Точка входа.** Любые публичные эндпоинты; особенно дорогие: auth (argon2id — намеренно дорогой), отправка уведомлений, вебхуки.

**Механизмы защиты.**
- **Rate limiting в Redis** — на auth жёсткий, на остальном API — базовый настраиваемый лимит per-tenant/per-IP.
- Лимит размера тела запроса; **обязательная пагинация** списков с потолком `page_size`.
- **Таймауты, retry с экспоненциальным backoff и circuit breaker** на все внешние вызовы — зависший Payme/Eskiz не съедает воркеры и пул БД; система деградирует, а не падает.
- Очереди arq: повторы с потолком, dead letter — ядовитая задача не крутится вечно; `/metrics` отдаёт глубину очередей, латентность и коды HTTP — всплеск виден до падения.
- `/health`, `/ready` + stateless-приложение — оркестратор перезапускает/масштабирует безболезненно.
- Суточный потолок исходящих SMS per-tenant — защита от «денежного» DoS. Это простой анти-абьюз-счётчик в Redis, а не тарифные квоты/usage metering из бэклога; включение в Фазу 3 — открытый вопрос №13 (рекомендация — включить).

**Остаточный риск.** Волюметрический L3/L7 DDoS на уровне приложения не решается — нужен внешний слой. Но зарубежный CDN (Cloudflare и т.п.) терминирует TLS и пропускает весь трафик с ПДн через зарубежную инфраструктуру, что конфликтует с дефолтом «прод-контур в РУз» (§3.3) — выбор слоя (локальный reverse-proxy / анти-DDoS провайдера в РУз vs зарубежный CDN с юр-оценкой) — открытый вопрос №12. Rate limit по IP слаб против ботнетов — принимаем для v1.

### V10. Инсайдер / скомпрометированный админ

**Кто и как.** Сотрудник подрядчика (мы же и есть подрядчик для клиентов), админ тенанта или атакующий, захвативший админ-аккаунт: выгрузка ПДн, правка платёжных статусов, заметание следов.

**Точка входа.** Админ-API; прямой доступ к серверу/БД в проде.

**Механизмы защиты.**
- **Append-only аудит**: кто, что, когда, откуда (request_id, tenant_id, user_id, IP) для всех значимых действий; на уровне БД runtime-роли имеют на `audit_log` только `SELECT, INSERT` — `UPDATE` не выдан никому, кроме владельца схемы (отсутствующего в runtime), `DELETE` — только выделенной ретенционной роли `app_retention` (схема §2.5) — подчистить следы через приложение нельзя.
- **Кросс-тенантный доступ платформенного админа — не обход RLS, а явный контролируемый путь**: вездеход-роли в runtime нет; операции поверх тенантов идут либо в явно установленном контексте целевого тенанта после проверки платформенного права, либо через исчерпывающий список системных операций `app_maintenance` (схема §3.4). Каждое использование платформенного права — отдельная запись в аудите; сам путь покрыт негативным тестом (доступен только носителю платформенного права). Носитель платформенной роли (флаг vs служебный тенант) — открытый вопрос №7 документа интерфейсов.
- RBAC по принципу минимальных привилегий: platform-admin ≠ tenant-owner; «суперкнопки» (доступ ко всем тенантам) — отдельное явно выдаваемое право, каждое использование — в аудит.
- 2FA для админских аккаунтов (обязательность — см. Открытые вопросы, №4).
- **Секреты через API не читаются обратно**: платформенные секреты живут только в env — их в БД и админ-API нет вовсе; per-tenant конфиги каналов уведомлений — write-only (задать/заменить можно, прочитать в открытом виде нельзя; в БД — только шифртекст, A6). Если по вопросу №10 будет принята per-tenant-модель мерчантов — их хранение оформляется тем же паттерном отдельным ADR.
- Захваченный админ ограничен плоскостью API: прямого SQL и shell админка не даёт by design.

**Остаточный риск.** Админ с доступом к серверу/БД (ops) находится вне модели приложения — организационные меры: доступ к прод-БД по отдельным именным учёткам, бэкапы шифруются (A10), доступ логируется. Уходит в чек-лист развёртывания (Фаза 5). Аудит защищён от приложения, но не от superuser БД — честно принимаем.

## 3. Закон РУз «О персональных данных» (ЗРУ-547)

> Сверено с открытыми источниками на 2026-07-05 (веб-поиск выполнялся).
> Это **инженерная** трактовка для проектирования; пункты, требующие
> юриста, помечены **[юрист]**. Реквизиты нормативных актов приведены
> по открытым источникам и до юр-подтверждения не считаются
> установленным фактом — особенно перед тиражированием в клиентские
> памятки (см. вопрос №9). Источники — в §3.6.

### 3.1. Правовая база и что изменилось

- Базовый закон: **ЗРУ-547 от 02.07.2019** «О персональных данных», в силе с 01.10.2019.
- Поправки 2021 г. (ст. 27¹, в силе с апреля 2021): жёсткая локализация — сбор, систематизация и хранение ПДн граждан РУз только на технических средствах, физически размещённых в РУз и зарегистрированных в Госреестре баз персональных данных.
- **Поправки марта 2026 г. (по открытым источникам — закон № ЗРУ-1125 от 26.03.2026, в силе с 27.03.2026; реквизиты подлежат юр-подтверждению [юрист])** — режим смягчён. Обязательная локализация в РУз теперь только для трёх категорий: **биометрические данные**, **генетические данные**, **данные физлиц — пользователей услуг операторов телекоммуникаций, работающих в РУз**. Прочие ПДн можно хранить/обрабатывать за рубежом при выполнении хотя бы одного условия: (а) иностранное государство признано обеспечивающим адекватный уровень защиты; (б) оператор применяет стандартные договорные условия / binding corporate rules, соответствующие требованиям уполномоченного органа; (в) оператор соблюдает международные стандарты управления и хранения ПДн из утверждённого перечня. **[юрист]** — подзаконная практика по этим условиям (перечни, формы) на момент написания молодая; перед выносом данных за рубеж проверять актуальные акты.
- Регистрация: базы ПДн, подлежащие обязательному хранению в РУз, регистрируются в **Государственном реестре баз персональных данных**; уполномоченный орган — Государственный центр персонализации (подведомственность центра со временем менялась — актуальную принадлежность проверить по lex.uz **[юрист]**, вопрос №9(е)); надзор и блокировки — **Узкомназорат**. **[юрист]** — уточнить, требуется ли после поправок 2026 г. регистрация баз, не попадающих под обязательную локализацию.
- Практика принуждения реальна: за нарушения локализации в 2021–2022 гг. ограничивался доступ к TikTok, Twitter, VK и др.

### 3.2. Что в нашем ядре является ПДн

ПДн по закону — любая зафиксированная информация, относящаяся к определённому физлицу или позволяющая его идентифицировать. В ядре v1 это:

| Данные | Таблицы (по схеме Фазы 0) |
|---|---|
| ФИО, email, телефон | `users` |
| telegram chat_id, адреса доставки уведомлений | `notification_settings` (конфиги каналов), `notification_outbox.recipient` |
| IP-адреса, user_agent | `audit_log`, `refresh_tokens` (экран «мои сессии») |
| Привязка человек↔организация | `memberships`, `invitations` |
| Платёжные идентификаторы физлиц (телефон в транзакциях Payme/Click) | `payments.metadata`, `payment_webhooks.raw_body` |

Принципиально: **ядро v1 сознательно не собирает биометрические и генетические данные** — самая жёсткая категория локализации на шаблон не распространяется. Категория «данные пользователей услуг операторов телекоммуникаций» относится к операторам связи; типовые клиенты шаблона (e-commerce, SaaS, CRM) под неё не подпадают. **[юрист]** — подтвердить трактовку для конкретного клиента, если клиент близок к телеком-сфере.

### 3.3. Следствия для шаблона и клиентских проектов

1. **Хостинг — по умолчанию в РУз.** Несмотря на смягчение 2026 г., дефолт шаблона: прод-контур клиентского проекта (Postgres, Redis, приложение) — в дата-центре на территории РУз. Причины: условия для зарубежного хранения требуют юридической работы на каждый проект; практика регулятора — блокировки; клиентам это дешевле объяснимо. Зарубежный хостинг — осознанное исключение с юристом, не дефолт.
2. **Бэкапы = те же ПДн (актив A10).** Бэкапы БД хранятся в той же юрисдикции, что и основная БД, и шифруются at-rest; работоспособность восстановления регулярно проверяется (restore-тест — требование мастер-промпта к чек-листу Фазы 5). Копия «на всякий случай» в зарубежном S3 — это трансграничная передача со всеми последствиями. Пункт входит в чек-лист развёртывания (Фаза 5).
3. **Регистрация БД.** Для каждого клиентского проекта на старте решается вопрос регистрации базы ПДн в Госреестре (ответственный — клиент как собственник/оператор БД; мы даём памятку). **[юрист]** — типовая памятка клиенту готовится с юристом один раз и кладётся в docs/.
4. **Роли по закону.** Собственник/оператор базы ПДн — клиент; мы — разработчик/обслуживающая сторона. Разграничение ответственности фиксируется договором. **[юрист]**.
5. **Согласие и права субъекта.** Регистрация пользователя должна опираться на законное основание обработки (согласие/договор); субъект вправе требовать сведения об обработке и уничтожение данных. Для шаблона: удаление/анонимизация аккаунта не должна ломать append-only аудит и платёжную отчётность — см. Открытые вопросы, №3 (retention-политика).

### 3.4. Трансграничные потоки: что уходит наружу и как минимизируем

| Сервис | Где серверы | Что уходит | Минимизация |
|---|---|---|---|
| Payme, Click | РУз | суммы, id платежей, телефон плательщика | штатный локальный поток — ок |
| Eskiz | РУз | телефон + текст SMS | локальный провайдер — ок; в текстах не слать лишние ПДн |
| Telegram Bot API | за рубежом | chat_id + текст уведомления | шаблоны уведомлений не содержат лишних ПДн (не класть в текст паспортные данные, полные адреса); отправка — по подписке пользователя (его выбор канала) |
| SMTP | зависит от релея | email + текст письма | предпочитать локальный/самостоятельный релей; та же дисциплина шаблонов |
| Sentry (SaaS) | за рубежом | стектрейсы, контекст ошибок | **scrubbing обязателен**: `send_default_pii=False`; `include_local_variables=False` — стектрейсы без локальных переменных (в locals ПДн/секреты лежат под произвольными именами, denylist их не ловит); `before_send` вырезает email/телефоны/токены/тела запросов; в событие идут только `request_id`, `tenant_id`, `user_id` (внутренние UUID — псевдонимизированные идентификаторы, не ФИО/контакты); альтернатива — self-hosted Sentry в РУз (см. Открытые вопросы, №1) |

Правило шаблона: **наружу уходит псевдоним (UUID), а не персона**. По `user_id` в Sentry нельзя установить личность без доступа к нашей БД. **[юрист]** — это инженерная минимизация, а не юридическая индульгенция: для оператора UUID остаётся идентифицируемым, и квалификация псевдонимизированных идентификаторов при передаче за рубеж требует подтверждения — включено в пакет юр-проверки, вопрос №9(д).

### 3.5. Что попадает в артефакты последующих фаз

- Фаза 1: Sentry-скраббер (`send_default_pii=False`, `include_local_variables=False`, `before_send`) + тест «PII не уходит в событие»; логи без ПДн-полей, где они не нужны; конвенция «arq-задача получает tenant_id в аргументах» в базовых классах shared/.
- Фаза 2: механика транзакционно-локального tenant-контекста + обязательные RLS-тесты (fail closed, отсутствие утечки через пул); минимальный аудит-интерфейс (при принятии вопроса №11); reset-токены и локаут 2FA по §V2.
- Фаза 3: шаблоны уведомлений проходят ревью на минимизацию ПДн; журнал вебхуков; тесты конверсии единиц адаптеров.
- Фаза 5: чек-лист развёртывания — хостинг в РУз, шифрование бэкапов, их юрисдикция и **регулярный restore-тест**, DDoS-слой (вопрос №12), памятка по регистрации БД, ротация секретов и ключа JWT.

## 5. Открытые вопросы

1. **Sentry: SaaS или self-hosted?** SaaS-Sentry — серверы за рубежом; при строгом scrubbing'е туда уходят только псевдонимы (UUID). Допустимость передачи псевдонимов — юридический вопрос (см. №9(д)), инженерно поток минимизирован. Альтернатива — self-hosted Sentry в РУз (плюс: ноль трансграничных потоков; минус: своя эксплуатация). **Рекомендация:** v1 — SaaS + обязательный scrubbing, self-hosted — опция конфигурации (DSN и так из env, коду всё равно). Меняется: только пункт чек-листа развёртывания.
2. **Дефолт хостинга после смягчения закона (03.2026).** Оставить жёсткий дефолт «прод и бэкапы — в РУз» или разрешить зарубежный хостинг как равноправный вариант с юр-оговорками? **Рекомендация:** оставить «в РУз» дефолтом, зарубежный — только по явному решению клиента с юристом. Меняется: текст чек-листа Фазы 5 и памятки клиенту.
3. **Retention и удаление ПДн vs append-only аудит.** Субъект вправе требовать уничтожения данных, но аудит и платёжная история должны сохраняться. Вариант A: анонимизация (`users` затирается, `user_id` в аудите остаётся как «мёртвый» UUID). Вариант B: полное каскадное удаление (в схеме §2.1 — жёсткое удаление user с чисткой ПД за пределами каскада). **Рекомендация:** A (анонимизация) как операция сервиса auth, согласовать с механикой удаления из схемы (§2.1, откр. вопрос №6 схемы) в Фазе 2. Меняется: схема auth и один эндпоинт. **Решение владельца (2026-07-06): вариант B — жёсткое удаление по механике схемы §2.1 (ОВ-02); рекомендация этого документа отклонена. Юридическая достаточность проверяется юристом в пакете №9.**
4. **Обязательная 2FA для админов.** Принуждать ли platform-admin и tenant-owner к TOTP (нельзя войти без настройки 2FA)? **Рекомендация:** да для platform-admin, для tenant-owner — настройка тенанта (по умолчанию выключено). Меняется: одна проверка в auth-флоу Фазы 2.
5. **Анти-стаффинг сверх rate limit (V2).** CAPTCHA/Proof-of-Work и проверка паролей по базам утечек — в v1 или бэклог? **Рекомендация:** бэклог; в v1 достаточно rate limit (включая per-account ключ) + локаут 2FA + уведомления о неудачных входах. Меняется: состав Фазы 2.
6. **gitleaks в CI (V6).** Мастер-промпт фиксирует pip-audit/bandit/trivy; добавить ли секрет-сканер gitleaks отдельным джобом? **Рекомендация:** да — дёшево и закрывает главный самострел шаблона. До решения строка «скан секретов» в §4 выполняется поиском по паттернам ключей из самопроверки фазы. Меняется: одна строка в CI Фазы 1. **Решение (2026-07-06): да, gitleaks добавлен в CI Фазы 1 (ОВ-30).**
7. **Reconciliation платежей (V4).** Регулярная сверка наших статусов с выписками Payme/Click: в v1 — ручная процедура в docs или фоновая arq-задача? **Рекомендация:** v1 — документированная ручная процедура; автоматизация — бэклог. Меняется: состав Фазы 3.
8. **Алгоритм подписи JWT.** HS256 (симметричный, проще) или RS256/EdDSA (асимметричный — верификация без секрета, полезно будущему tg-bot-шаблону, который будет проверять токены). **Рекомендация:** v1 — HS256 (монолит, один владелец секрета), решение зафиксировать в ADR с планом миграции на асимметрию при появлении внешних верификаторов. В ADR обязательна оговорка про alg-confusion: при миграции на RS256 верификатор принимает строго ожидаемый алгоритм из конфига (см. V3), иначе публичный RSA-ключ может быть использован как HMAC-секрет. Меняется: конфиг auth Фазы 2.
9. **[юрист] Пакет юридической проверки** (один заход к юристу, результаты — в docs/): (а) регистрация баз ПДн в Госреестре после поправок 2026 — для кого обязательна; (б) условия зарубежного хранения (адекватные страны, формы SCC/BCR); (в) типовая памятка клиенту о ролях собственника/оператора БД ПДн; (г) трактовка категории «данные пользователей услуг операторов телекоммуникаций» для клиентов, смежных с телекомом; (д) квалификация псевдонимизированных идентификаторов (внутренние UUID в Sentry) при передаче за рубеж; (е) подтверждение реквизитов и содержания ЗРУ-1125 и актуальной подведомственности уполномоченного органа перед тиражированием в памятки.
10. **Модель merchant-аккаунтов: один на развёртывание или per-tenant.** Биллинг ядра v1 — подписки тенантов в пользу владельца развёртывания, поэтому естественный дефолт: один merchant-аккаунт Payme/Click на инсталляцию, ключи только в env (A6). Альтернатива — per-tenant мерчанты (каждый тенант принимает платежи на свой аккаунт, сценарий маркетплейса): тогда ключи становятся данными — шифрованное хранение в БД (по паттерну `notification_settings.config_encrypted`), write-only-операции, ротация, выбор ключа проверки вебхука по платежу. **Рекомендация:** v1 — один мерчант на развёртывание (env); per-tenant — отдельное решение с ADR при реальном спросе клиента. Меняется: A6, V4, V10; при per-tenant — схема billing. **Решение (2026-07-06): один мерчант на развёртывание, креды только в env (ОВ-05).**
11. **Минимальный аудит раньше Фазы 4.** По PLAN аудит строится в Фазе 4, но заявлен митигацией в Фазе 2 (неуспешные входы, reuse refresh) и Фазе 3 (след вебхуков). Вариант A: поднять минимальный core/audit уже в Фазе 2 — таблица `audit_log` (append-only) + узкий `AuditService.record()`; Фаза 4 добавляет чтение, админ-экраны и полный охват действий. Вариант B: до Фазы 4 каждая фича ведёт собственный журнал (журнал входов в auth, журнал вебхуков в billing), Фаза 4 добавляет общий аудит поверх. **Рекомендация:** A — интерфейс тривиален, а два поколения журналов и их миграция дороже. Меняется: состав Фазы 2 (одна таблица + сервис), примечание в §4.
12. **DDoS-слой vs локализация данных.** Волюметрический DDoS решается только внешним слоем, но зарубежный CDN (Cloudflare и т.п.) терминирует TLS и пропускает весь трафик с ПДн через зарубежную инфраструктуру — конфликт с дефолтом «прод-контур в РУз» (§3.3). Вариант A: локальный reverse-proxy + анти-DDoS-защита хостинг-провайдера в РУз. Вариант B: зарубежный CDN после юр-оценки **[юрист]**. **Рекомендация:** A как дефолт чек-листа Фазы 5; B — осознанное исключение вместе с вопросом №2. Меняется: чек-лист Фазы 5.
13. **Суточный потолок исходящих SMS (анти-абьюз, V9).** Механизм не зафиксирован мастер-промптом и граничит с квотами тарифов из бэклога (модуль saas). Вариант A: простой суточный лимит-счётчик per-tenant в Redis в Фазе 3 + тест `test_sms_daily_cap_enforced` — это анти-абьюз-контроль против «денежного» DoS (Eskiz — прямые расходы), а не usage metering. Вариант B: убрать из механизмов V9 и отложить в бэклог к saas. **Рекомендация:** A — счётчик тривиален, а риск прямых денежных потерь реален. Меняется: состав Фазы 3, строка V9 в §4.
14. **Транспорт refresh-токена (V3).** Вариант A: refresh в теле ответа (bearer) — единый механизм для web/mobile/ботов; cookie нет ⇒ нет CSRF-поверхности на `/auth/refresh`; минус — хранение на стороне клиента (для web-SPA выше XSS-экспозиция). Вариант B: httpOnly+Secure cookie — недоступен XSS, но добавляет CSRF-поверхность (обязательны `SameSite` + анти-CSRF-токен) и требует отдельной ветки для не-браузерных клиентов. **Рекомендация:** v1 — A (шаблон API-first, клиенты разнотипные); web-клиентам рекомендовать BFF/httpOnly-обёртку на их стороне; cookie-режим — расширение по спросу, только вместе с CSRF-защитой. Меняется: auth-флоу Фазы 2 и документация для клиентов.
