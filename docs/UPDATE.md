# Процедура обновления клиентского проекта

> Скелет (Фаза 1). Финальная версия с чек-листом security-раскатки по
> реестру и hotfix-процедурой — в Фазе 5. Основание: ADR-0009,
> `docs/phase0/04-update-strategy.md` §4.2.8.

## Предусловия

- Проект создан клоном шаблона: remote `template` существует
  (`git remote -v`), история общая. Копия файлов без истории —
  антипаттерн, merge работать не будет.
- `CUSTOMIZATIONS.md` в корне проекта актуален (все помеченные
  `# CUSTOM(<project>)` правки шаблонных путей задекларированы).

## Шаги

1. `git fetch template --tags`
2. Прочитать в `CHANGELOG.md` шаблона раздел нового релиза, особенно
   **Upgrade notes** (и всех промежуточных релизов, если проект отставал).
3. `git merge --no-ff vX.Y.Z`
   - Конфликтов нет → шаг 4.
   - Конфликт в шаблонном пути → это сигнал: кастомизацию выносим из ядра
     (своя фича-папка, конфиг, событие шины) либо отправляем upstream в
     шаблон. Правка по месту — крайний случай, только с маркером
     `# CUSTOM(<project>): <причина>` и записью в `CUSTOMIZATIONS.md`.
   - `merge --squash` и rebase поверх влитых тегов **запрещены** —
     уничтожают общего предка (ловится CI-проверкой предка).
4. `git push origin vX.Y.Z` — тег обязан попасть в origin: по нему работает
   CI-джоба `template-drift`.
5. `uv sync`
6. `make migrate` (= `alembic upgrade heads` — все ветки) на dev-БД.
7. `make lint && make test`
8. Smoke: `GET /health`, `GET /ready` на dev-стенде.
9. push → deploy по процедуре проекта.
10. Отметка в реестре проектов: версия, дата merge, дата deploy
    (merge ≠ deploy; реестр — единственный источник ответа «фикс в проде»).

## Появится в Фазе 5

- Чек-лист «раскатка security-фикса по всем проектам реестра».
- Hotfix-процедура для отставшего проекта (cherry-pick `[SECURITY]`-коммитов
  с обязательным догоняющим merge в течение 14 дней).
- Шаблон реестра проектов + read-only drift-check-скрипт.
