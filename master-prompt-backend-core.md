## АРХИТЕКТУРНЫЕ РЕШЕНИЯ (зафиксированы, не обсуждаются)

1. **Модульный монолит.** Не микросервисы. Любой модуль можно выпилить
   или вынести в отдельный сервис позже.
2. **Стек:** Python 3.12+, FastAPI, SQLAlchemy 2.x (async), Alembic,
   PostgreSQL 16, Redis (кеш + очереди через arq), Docker Compose.
   Пакетный менеджер — uv. Только зрелые популярные библиотеки. Никаких
   экзотических зависимостей — каждая новая требует обоснования.
3. **Мультитенантность с первого дня:** колонка `tenant_id` во всех
   бизнес-таблицах (утверждённое исключение Фазы 0, ОВ-01: глобальные
   auth-таблицы `users`/`user_totp`/`user_recovery_codes`/`refresh_tokens`
   и глобальные справочники) + автоматическая фильтрация на уровне репозитория
   (невозможно случайно прочитать чужого тенанта). Row-level security
   в Postgres как вторая линия обороны.
4. **Трёхуровневый конструктор: ядро → модули → фичи.**
   - **Ядро** — то, что нужно каждому проекту: auth, тенанты, биллинг,
     уведомления, аудит, каркас админки.
   - **Модуль** — предметная область (commerce, позже crm, saas).
     Включается конфигом: `ENABLED_MODULES = ["commerce"]`. Выключен —
     его роутов, таблиц и тестов в проекте нет.
   - **Фича** — вертикальный срез внутри модуля (products, cart,
     orders). Живёт в своей папке целиком и переносится в клиентский
     проект копированием папки. Анатомия — в разделе ниже.
5. **Правила связей.** Зависимости направлены только «вниз»: фича →
   ядро → shared. Горизонтально (фича → фича, модуль → модуль) — только
   через публичные сервисные интерфейсы или события. Чтение чужих
   таблиц запрещено всегда, без исключений. Каждая горизонтальная связь
   декларируется в feature.toml фичи и проверяется автоматически — на
   старте приложения и тестом в CI.
6. **Событийная шина** в shared/: лёгкий in-process pub/sub; обработчики,
   требующие надёжности, уходят фоново через arq. Модули реагируют друг
   на друга, не зная друг о друге: commerce публикует
   `commerce.order.created` — будущий crm подписывается.

### Структура

```
backend-core/
```

## АНАТОМИЯ ФИЧИ

Каждая фича — самодостаточная папка:

```
modules/commerce/products/
├── feature.toml    # машиночитаемый манифест связей (обязателен)
├── models.py       # таблицы фичи; владеет ими эксклюзивно
├── schemas.py      # Pydantic-схемы границы API
├── service.py      # бизнес-логика; публичный интерфейс фичи
├── router.py       # эндпоинты /api/commerce/products
├── admin.py        # админ-экраны фичи; регистрируются в core/admin сами
├── migrations/     # миграции только этой фичи
├── tests/          # переезжают вместе с фичей
└── README.md       # инструкция для человека (обязательна)
```

**feature.toml — машиночитаемый манифест**, единственный источник
правды о связях фичи:

```toml
name = "commerce.cart"
description = "Корзина покупателя"
requires_features = ["commerce.products"]  # без них фича не работает
requires_core = ["auth", "tenants"]
owns_tables = ["commerce_carts", "commerce_cart_items"]
publishes_events = ["commerce.cart.checked_out"]
listens_events = []
```

Манифест — не документация, а механизм. На нём работают три вещи:
1. **Автодискавери и проверка на старте.** Модуль собирает себя из
   фич-папок, физически присутствующих на диске. Если фича есть, а её
   requires — нет, приложение падает на старте с понятной ошибкой:
   «commerce.cart требует commerce.products, которая не установлена».
2. **Перенос с зависимостями.** tools/add-feature читает манифест и
   тянет цепочку requires целиком: попросил cart — получишь cart +
   products (с подтверждением).
3. **Тест честности манифеста в CI.** Тест обходит импорты каждой фичи
   и падает, если фича использует то, чего нет в её requires. Манифест,
   который врёт, — красный CI.

**README.md — инструкция для человека:** назначение (2–3 строки),
публичный интерфейс фичи (что можно звать снаружи), шаги подключения
в новый проект, типовые кастомизации.

Правило нарезки: независимое — отдельной фичей, опциональное — отдельной
фичей. Примеры: `products` переносится в одиночку; `orders` объявляет
`requires: products` и переносится парой; картинки товаров — будущая
фича `product-images` (придёт вместе с core/files из бэклога), а не
часть `products`.

## ИНСТРУКЦИЯ МОДУЛЯ

У каждого модуля — свой README.md. Для модулей ядра: зачем модуль,
публичный интерфейс (сервисы и события), как расширять. Для
бизнес-модулей README — это меню конструктора, отвечающее на вопрос
«что отсюда можно собрать под заказ клиента»:

- из каких фич состоит модуль (по строке на фичу);
- карта зависимостей фич:

  | Фича     | Требует  |
  |----------|----------|
  | products | —        |
  | cart     | products |
  | orders   | products |

- рецепты сборки под типовые заказы: «минимальный магазин = products +
  orders», «магазин с корзиной = products + cart + orders»;
- как добавить в модуль новую фичу (пошагово).

## ИНЖЕНЕРНЫЕ СТАНДАРТЫ (обязательны для каждой строки кода)

### Безопасность (по мотивам OWASP ASVS)
- Пароли: argon2id. Токены: короткоживущий access JWT + refresh с
  ротацией и отзывом. Секреты — только из переменных окружения,
  в репозитории — `.env.example` с фиктивными значениями.
- Вся валидация входных данных — Pydantic-схемами на границе API.
  Внутрь системы не проходит ни один невалидированный байт.
- Никакой конкатенации SQL. Только параметризованные запросы через ORM.
- Rate limiting на Redis: лимиты на auth-эндпоинтах, экспоненциальные
  задержки + блокировка при переборе.
- Права проверяются на уровне сервиса, не только роутера. Механика
  «эндпоинт без декларации прав не поднимается»: обязательная
  зависимость `require_permission(...)`; на старте приложение обходит
  все роуты и падает, если у роута нет маркера прав; на это есть тест.
- Строгий CORS по умолчанию (белый список origins из конфига) +
  security headers middleware (HSTS, X-Content-Type-Options,
  X-Frame-Options, Referrer-Policy).
- Логи никогда не содержат: пароли, токены, полные номера карт, коды 2FA.

### Надёжность и эксплуатация (12-factor + SRE-практики)
- Конфигурация только через окружение. Приложение stateless, состояние —
  в Postgres/Redis.
- Структурированные JSON-логи с request_id, tenant_id, user_id в каждой
  записи. Здоровье: `/health` (liveness) и `/ready` (readiness,
  проверяет БД и Redis).
- Sentry с первого дня (DSN из окружения; пусто — выключен).
- `/metrics` в формате Prometheus: латентность и коды HTTP, глубина
  очередей arq.
- Все внешние вызовы (платёжки, SMS, Telegram) — с таймаутами, повторами
  (экспоненциальный backoff) и circuit breaker. Внешний сервис упал —
  система деградирует, а не падает.
- Платёжные операции идемпотентны (idempotency key). Вебхуки от платёжек
  переживают повторную доставку без дублей.
- Фоновые задачи — через очередь (arq), с повторами и dead letter.
- Миграции обратимы (у каждой есть рабочий downgrade).

### Качество кода
- Типизация везде, mypy в strict-режиме проходит без ошибок.
- Линтер ruff, форматирование автоматическое. Настрой pre-commit.
- Тесты: pytest. Пирамида — много юнит-тестов на бизнес-логику,
  интеграционные на каждый эндпоинт (testcontainers с реальным
  Postgres), минимум моков. Покрытие ядра ≥ 85%, проверяется в CI.
- Обязательные негативные тесты: чужой тенант не читается, права
  enforced, невалидный ввод отклоняется, вебхук с неверной подписью
  отвергается.
- CI (GitHub Actions): lint → mypy → tests → pip-audit + bandit →
  build образа (+ trivy-скан). Красный CI = не мержим.
- Ключевые архитектурные решения — файлами в docs/adr/ (контекст,
  решение, последствия). Раздел «Архитектурные решения» этого промпта —
  это ADR №1–6, оформи их в Фазе 1.
- Каждый модуль — README-инструкция по разделу «Инструкция модуля»;
  каждая фича — feature.toml + README по разделу «Анатомия фичи».

### Локализация (i18n)
- Все пользовательские тексты (ошибки API, шаблоны уведомлений) — через
  каталоги сообщений. Старт: ru и uz (латиница); uz-кириллица — третьим
  каталогом по требованию клиента.
- Определение локали: заголовок запроса → настройка пользователя →
  настройка тенанта → ru по умолчанию.

### Деньги
- Суммы — только целые числа в минимальных единицах валюты + поле
  `currency` (ISO 4217; по умолчанию UZS). Никаких float. Экспонента
  валюты — справочником (UZS = 0, USD = 2), чтобы commerce не был
  прибит к UZS и Stripe из бэклога встал без миграций.

## САМОПРОВЕРКА В КОНЦЕ КАЖДОЙ ФАЗЫ

Прежде чем отчитаться, проверь и явно подтверди каждый пункт:
1. `make lint`, `make test` — зелёные. Если нет — чини, не отчитывайся.
2. Ни одного TODO/заглушки/pass в продакшен-путях. Если что-то не
   реализовано — это сказано прямо в отчёте, а не спрятано.
3. Новые таблицы имеют tenant_id и покрыты тестом изоляции тенантов.
4. Новые эндпоинты имеют декларацию прав и негативный тест на 403.
5. Новая фича имеет feature.toml и README; тест честности манифестов
   зелёный (импорты соответствуют requires); чужие таблицы не читаются;
   README модуля обновлён (карта фич и рецепты сборки).
6. Секретов в коде нет (проверь поиском по паттернам ключей).

## ЧЕГО НЕ ДЕЛАТЬ

- Не изобретай абстракции «на будущее», которых не требует текущая фаза.
  Лучший код — скучный и очевидный.
- Не строй ничего из бэклога и заготовок под него — даже «пустые папки
  на потом».
- Не позволяй фиче лезть в таблицы другой фичи или модуля — даже
  «временно, потом починим».
- Не добавляй зависимости без необходимости и обоснования.
- Не глотай исключения. Не пиши `except Exception: pass` никогда.
- Не оптимизируй преждевременно; но и не делай N+1 запросов — это не
  оптимизация, это гигиена.
- Если требование неоднозначно — задай вопрос, не угадывай молча.
  Одно принятое молча неверное допущение дороже пяти вопросов.
